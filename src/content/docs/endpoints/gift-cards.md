title: 'Gift Cards Endpoint Group'
description: 'Coverage notes and fidelity boundaries for Gift Cards Endpoint Group.'

Gift-card reads and lifecycle mutations are modeled as local-first Admin GraphQL support. Supported gift-card mutations update normalized in-memory state and retain the original raw request for commit replay, without sending writes or notification side effects to Shopify during normal runtime handling.

## Current support and limitations

`src/operation_registry.rs` routes the gift-card roots below through local Rust dispatch. The support claims are based on runtime handlers, checked-in parity specs, tests, and fixtures; registry presence alone is not support.

### Supported roots

Overlay reads:

- `giftCard(id:)`
- `giftCards(...)`
- `giftCardsCount(...)`
- `giftCardConfiguration`

Local staged mutations:

- `giftCardCreate(input:)`
- `giftCardUpdate(id:, input:)`
- `giftCardCredit(id:, creditInput:)`
- `giftCardDebit(id:, debitInput:)`
- `giftCardDeactivate(id:)`
- `giftCardSendNotificationToCustomer(id:)`
- `giftCardSendNotificationToRecipient(id:)`

### Local read behavior

- Gift-card reads resolve from effective gift-card state: staged records overlay restored or hydrated base records, and snapshot-only reads never invent merchant gift cards from captured IDs.
- Snapshot mode returns `null` for unknown `giftCard(id:)`, an empty `giftCards` connection, and `{ count: 0, precision: "EXACT" }` for `giftCardsCount` when no records are present.
- LiveHybrid mode forwards unstaged `giftCard(id:)`, `giftCards`, and `giftCardsCount` reads upstream so existing merchant gift cards remain visible. Successful upstream `giftCardConfiguration` reads hydrate configuration into base state, and upstream gift-card payloads hydrate base gift-card state only when the selected payload contains the model fields needed for later local mutation validation. Staged gift cards replace matching upstream connection nodes, append locally created cards that match the query, and adjust upstream counts for staged/base query membership changes.
- `giftCards` uses the shared connection helpers for `nodes`, `edges`, selected `pageInfo`, stable synthetic cursors, `first`/`last`, `before`/`after`, `sortKey: ID`, and `reverse`.
- Local query filtering covers `id` terms, documented `status:enabled` / `status:disabled`, documented `balance_status:full` / `balance_status:partial` / `balance_status:empty` / `balance_status:full_or_partial`, populated-data filters for `created_at`, `expires_on`, `customer_id`, `recipient_id`, `source`, and `initial_value`, and unfielded code-fragment searches against locally tracked create codes plus visible `lastCharacters` / `maskedCode` values. Date and money filters use the shared search helpers for comparator handling, so range filters such as `created_at:>=2026-04-29`, `expires_on:<2028-01-01`, and `initial_value:>=5` narrow local results and preserve empty connection/count behavior when no local record matches. `customer_id` and `recipient_id` match either the full customer GID or the numeric tail, mirroring the captured Shopify search form.
- Source filtering is represented as local metadata instead of a selected GiftCard field because Shopify does not expose `source` on the GiftCard object in the captured schema. Locally staged `giftCardCreate` records are tagged as `source:api_client`, and the LiveHybrid hydrate path tags the captured Admin API-created gift card the same way; unknown source values such as `source:manual` return no local matches when the record is known to be API-created.
- Live evidence shows `updated_at` does not currently narrow results for the captured gift-card search path: an ID-filtered query with `updated_at:>=2099-01-01` still returned the live-created card. The local proxy therefore keeps `updated_at` in the unsupported/no-op bucket instead of claiming implemented date filtering for it. Fields such as `enabled`, `active`, and `last_characters` are also invalid search fields and leave Shopify results unfiltered with warnings.
- `giftCardConfiguration` exposes `issueLimit` and `purchaseLimit` money objects from normalized base state. When no configuration has been restored or hydrated, the proxy derives the configured limit amounts in the effective shop currency rather than using a fixed CAD fixture.
- In LiveHybrid cassette parity, existing upstream gift cards referenced by supported mutation roots are hydrated with a narrow `GiftCardHydrate` read before local staging. The hydrate response persists the prior gift card and configuration into base state, then `giftCardUpdate`, `giftCardCredit`, `giftCardDebit`, and `giftCardDeactivate` stay local-only for their lifecycle effects and downstream reads.

### Local mutation behavior

- All supported gift-card mutation roots first check the effective shop gift-card entitlement at `ShopRecord.entitlements.giftCards.enabled`. The entitlement defaults to enabled when shop state is unknown or older serialized state lacks the entitlement slice. When the entitlement is disabled, `giftCardCreate`, `giftCardUpdate`, `giftCardCredit`, `giftCardDebit`, `giftCardDeactivate`, `giftCardSendNotificationToCustomer`, and `giftCardSendNotificationToRecipient` return a null payload resource plus a single `userErrors` entry with `field: null`, message `Gift cards are unavailable on your plan.`, and no code. This branch runs before the normal trial-shop, missing-card, amount/currency, notification, and other validation ladders, and it does not hydrate gift cards, stage records, append transactions, or send runtime Shopify writes.
- `giftCardCreate(input:)` stages a new normalized gift card with a proxy-synthetic `GiftCard` GID, Shopify-like generated or normalized lower-case code echo, bullet-masked/last-character code metadata, initial value and balance, optional note/expiry/template/customer/recipient attributes metadata, a stored `notify` flag, and stable timestamps. Omitted optional `note`, `expiresOn`, `customerId`, `templateSuffix`, and `recipientAttributes` inputs stage and read back as `null`; supplied note, expiry, customer, template suffix, and recipient metadata round-trip from real staged or hydrated customer records. When the local schema accepts `input.notify`, create stores the supplied boolean and defaults to `true` when the key is omitted. Provided codes are normalized by stripping whitespace and dashes, then validated for 8-20 alphanumeric characters and uniqueness against locally tracked gift-card codes; on trial shops, `customerId` assignment returns `INVALID` before customer lookup and recipient assignment through `recipientAttributes` or top-level `recipientId` returns `INVALID` before staging. Missing `customerId` references return `CUSTOMER_NOT_FOUND` based on customer existence, not sentinel ID substrings, before code, recipient, and initial value model validations on non-trial shops. After the trial-assignment, customer, lower-bound, code, and recipient-attributes checks, create validates `initialValue` against a readable non-zero `giftCardConfiguration.issueLimit`; the limit error uses the hydrated/restored configuration currency, or the effective shop currency when no configuration is present.
- For released public Admin API 2026-04, captured schema introspection shows `GiftCardCreateInput.initialValue` is required and `initialAmount` is not exposed. The proxy applies the shared captured-schema input validator before gift-card staging, so missing `initialValue` and submitted `initialAmount` return top-level GraphQL errors (`missingRequiredInputObjectAttribute`, `argumentNotAccepted`, or variable `INVALID_VARIABLE`) and do not append mutation-log entries. The local-only `notify` extension remains accepted for disabled-notification runtime coverage because the public schema cannot record that branch directly.
- `giftCardUpdate(id:, input:)` stages note, expiry, template suffix, customer, and recipient attributes metadata changes against base or staged gift cards. Existing gift cards are hydrated before validation in LiveHybrid mode; on trial shops, touched `customerId` assignment returns `INVALID` before changed-customer lookup, and touched recipient assignment through `recipientAttributes` or top-level `recipientId` returns `INVALID` before recipient validation or staging. Its missing-arguments validation is key-based, matching Shopify: a present editable key such as `note`, `expiresOn`, or `templateSuffix` is accepted even when the supplied value already equals the current value, and the local update still touches `updatedAt`.
- `giftCardCreate(input:)` and `giftCardUpdate(id:, input:)` normalize `templateSuffix` by removing exactly one literal leading `gift_card.` prefix before staging. For example, `gift_card.birthday` is stored and returned as `birthday`, while unrelated prefixes are left unchanged. Mutation payloads and downstream `giftCard(id:)` / `giftCards(...)` reads all expose the normalized stored value.
- `giftCardCreate(input:)` and `giftCardUpdate(id:, input:)` assume the caller has the customer-read permissions needed for customer or recipient assignment. The proxy does not currently model OAuth access scopes from the incoming session token, so it cannot reproduce Shopify's narrowed-token permission error that fires before trial-shop and customer-existence checks when `customerId`, `recipientId`, or `recipientAttributes` is supplied without customer read access. With full-scope assumptions, the local validation ordering remains entitlement, trial-shop assignment, customer existence, value/code/recipient validation, and then staging.
- For both create and update, a literal or variable-bound `recipientAttributes` object must include `id`; missing nested `id` returns the captured top-level `missingRequiredInputObjectAttribute` error path and stages no local state. When `recipientAttributes` is present with an `id`, the referenced customer must exist in staged customer state or hydrate from upstream in LiveHybrid mode; otherwise the mutation returns `RECIPIENT_NOT_FOUND` on `["input", "recipientAttributes", "id"]` with message `Recipient could not be found`. Supplied blank `preferredName` or `message` values return `INVALID` on the matching field path with the captured ActiveModel blank message. `preferredName` is capped at 255 characters, `message` is capped at 200 characters, both text fields reject HTML tags, and `sendNotificationAt` must parse as an ISO datetime within now through now plus 90 days. These validation failures return field paths under `["input", "recipientAttributes", ...]`, use captured messages, and leave gift-card records, balances, and transactions unchanged. Public Admin GraphQL exposes `GiftCardUpdatePayload.userErrors` as generic `UserError` in the captured API versions, so live update captures provide field/message evidence while local runtime coverage keeps the typed `code` contract aligned with the rest of the gift-card model.
- `giftCardCredit(id:, creditInput:)` and `giftCardDebit(id:, debitInput:)` validate non-positive amounts, missing cards, invalid/future `processedAt`, expired cards, deactivated cards, mismatched input currency, and debit insufficient funds before staging. `giftCardCredit` also validates `current.balance + creditAmount` against a readable non-zero `giftCardConfiguration.issueLimit`; if the configuration is still unknown for a locally known card, the proxy hydrates it with a narrow read before evaluating the limit. The captured public Admin API 2025-01 branch returns `GIFT_CARD_LIMIT_EXCEEDED` on `["creditInput", "creditAmount", "amount"]` with message `The gift card's value exceeds the allowed limits.` and does not append a transaction. `giftCardDebit` decreases balance and did not surface `GIFT_CARD_LIMIT_EXCEEDED` in the captured debit-after-rejected-credit path. Validation failures return Shopify-like `userErrors` without appending transactions, changing balances, or minting synthetic identities. Successful transactions always use the card balance currency, preserve explicit valid `processedAt` inputs, otherwise use synthetic timestamps, preserve supplied `note` strings, return omitted transaction notes as `null`, and emit typed `GiftCardCreditTransaction` / `GiftCardDebitTransaction` payload typenames.
- Expired-card validation for `giftCardCredit`, `giftCardDebit`, `giftCardSendNotificationToCustomer`, and `giftCardSendNotificationToRecipient` compares `expiresOn` against the hydrated shop-local date, using the stored `Shop.timezoneOffsetMinutes` baseline. In LiveHybrid mode, boundary-sensitive existing gift cards hydrate the shop baseline with a narrow `StorePropertiesShopBaselineHydrate` read before validation when no shop is already present. If no shop timezone baseline is available, the proxy falls back to the UTC date and writes a debug log to stderr so conformance runs make the weaker assumption visible.
- `giftCardDeactivate(id:)` stages `enabled: false` plus a synthetic `deactivatedAt` timestamp and keeps downstream reads visible.
- Notification roots validate Shopify-like failure branches before returning a local payload: trial shops, unknown cards, disabled notifications, expired or deactivated cards, missing customer/recipient assignment, missing customer/recipient records, and missing customer/recipient email or phone contact information. Base-scoped notification failures such as trial-shop rejection, no customer, and no contact information return `field: null` with `INVALID` codes. Successful notification requests remain local short-circuits that append mutation-log entries but do not send customer-visible notifications at runtime. Recipient attributes are kept as local gift-card metadata; the original raw notification mutations are still retained for explicit commit replay. The stored `notify` flag is honored from local/snapshot state; public Admin GraphQL 2025-01 does not expose a `GiftCard.notify` field or `GiftCardCreate`/`GiftCardUpdate` `notify` input for live setup, so disabled-notification parity is local-runtime backed.
- Gift-card mutation `userErrors.__typename` follows the parent payload's schema type. Create emits `GiftCardUserError`; credit/debit emit `GiftCardTransactionUserError`; deactivate and notification mutations emit their mutation-scoped user-error types; update remains generic `UserError` because the public schema types `GiftCardUpdatePayload.userErrors` that way.

### Unsupported, registry-only, and validation-only coverage

- Notification roots are supported only as validation plus local acknowledgement/logging boundaries. The proxy cannot verify or emulate Shopify email delivery, template rendering, customer notification preferences, bounce handling, or recipient inbox state.
- Search-fidelity gaps include source values other than the captured Admin API `api_client` path and any undocumented search fields not backed by executable conformance evidence. Captured `updated_at` behavior is documented as a no-op for the selected API/version rather than a supported local date filter.
- Disabled-notification, mutation user-error code, trial-shop assignment, and entitlement-disabled scenarios are local-runtime backed when public Admin GraphQL schema/access prevents a live success-path fixture.

- post-balance-write advanced search behavior for `created_at`, `expires_on`, `customer_id`, `recipient_id`, `source:api_client`, `source:manual`, and `initial_value`
