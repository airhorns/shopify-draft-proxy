### Unsupported, registry-only, and validation-only coverage

- Customer payment method live success-path coverage remains blocked on
  `read_customer_payment_methods` / `write_customer_payment_methods` scopes and
  sensitive payment material; the supported local slice uses scrubbed
  local-runtime fixtures and normalized payment-method shells.
- Draft-order transfer during `customerMerge` is not claimed because downstream
  draft-order transfer was not captured through a customer/draft-order read.
  Gift cards, discounts, and other attached resources also remain deferred until
  fixtures capture their non-empty downstream behavior.

### Behavior notes

- Customer-domain state deliberately stays narrower than the product model, but it is still normalized.
- `CustomerRecord` carries scalar/detail fields plus `taxExemptions` as a separate list from the boolean `taxExempt`, and `dataSaleOptOut` for the privacy-domain `dataSaleOptOut` mutation's downstream read effect.
- Customer-owned metafields use the shared owner-metafield overlay keyed by the
  customer GID. `customerCreate(input.metafields)` and
  `customerUpdate(input.metafields)` stage every submitted metafield, preserve
  distinct `namespace` / `key` rows, and project aliased `customer.metafield(...)`
  selections plus `customer.metafields(...)` connections from that owner store
  in both mutation payloads and downstream customer reads. The
  `customerSet(identifier.customId)` path also stages and preserves the
  identifier metafield when a valid CUSTOMER `id` definition exists. The
  metafields connection uses shared cursor windowing and honors `reverse`.
- `customerByIdentifier(identifier:)` resolves from the same effective normalized customer graph as `customer(id:)` and `customers`, including staged customer creates/updates and hydrated live-hybrid customers.
- In LiveHybrid mode, staged customer reads hydrate the relevant upstream/base
  customer slice and overlay staged customer records instead of replacing the
  live catalog with staged-only data. Staged records win by ID, customer
  tombstones suppress matching upstream rows, singular/identifier misses can
  still resolve from upstream, and list/count reads compute search, sort,
  reverse, cursor windows, and count payloads from the effective customer set.
- `Customer.orders` reads from the staged per-customer order index when local order writes or customer merge/order-customer changes affect the relationship. That staged connection applies the supported local order `query:`, `sortKey`, `reverse`, and cursor windowing before projection; hydrated inline order pages remain projected from their captured page when no staged order relationship exists.
- For locally staged overlay reads, `customers(query:)` and `customersCount(query:)`
  share the same staged predicate semantics for the supported local search slice.
  The staged query parser supports bare text, implicit `AND`, explicit `OR`,
  parentheses, unary `-`, and common keyed predicates including `email:`,
  `first_name:`, `last_name:`, `country:`, `state:`, `tag:`, `id:`, `phone:`,
  `verified_email:`, `created_at:`, `updated_at:`, and `orders_count:`. Unknown
  keyed predicates remain unsupported staged-search terms and do not narrow
  staged customer rows, so the proxy avoids turning an unimplemented Shopify
  search key into an empty local result set. Captured live `customersCount(query:)`
  warning behavior remains separate and must not be generalized into broader
  count-query support without conformance evidence.
- In Admin GraphQL 2026-04, customer marketing consent readback is conformance-backed through `Customer.defaultEmailAddress` and `Customer.defaultPhoneNumber` marketing fields. Staged email/SMS consent updates keep those default contact fields and the compatibility `emailMarketingConsent` / `smsMarketingConsent` serializers aligned for downstream reads.
- `customerEmailMarketingConsentUpdate` and `customerSmsMarketingConsentUpdate` model the captured validation matrix: GraphQL-variable failures for missing/null nested consent payloads, null required `marketingState`, invalid enum values, invalid `DateTime`, and unsupported input fields; resolver-level rejection for `NOT_SUBSCRIBED` / `REDACTED` input states; email `INVALID` rejection; `SUBSCRIBED` without `marketingOptInLevel` returning `MISSING_ARGUMENT`; `PENDING` requiring `CONFIRMED_OPT_IN`; future timestamp userErrors; the email no-default-email no-op branch; and the SMS no-default-phone userError branch. Dedicated 2026-04 parity specs replay the disallowed-state cases directly: email `NOT_SUBSCRIBED` / `REDACTED` / `INVALID` and SMS `NOT_SUBSCRIBED` / `REDACTED` reject with top-level `INVALID`, while SMS `INVALID` is a GraphQL enum validation failure (`INVALID_VARIABLE`) before resolver execution. Validation failures do not mutate normalized customer consent state.
- Email and SMS validation payload shapes intentionally differ where Shopify differs: unknown email customers report `field: ["input", "customerId"]` / `code: "INVALID"`, unknown SMS customers report null field/code, future timestamp email failures return the unchanged customer payload, and future timestamp/SMS no-phone failures return `customer: null`.
- Customer-owned addresses live in normalized `customerAddresses` state. Staged `customerAddressCreate`, `customerAddressUpdate`, `customerAddressDelete`, and `customerUpdateDefaultAddress` mutate that address graph locally and keep `Customer.defaultAddress` synchronized with the selected default row.
- `Customer.addresses` and `Customer.addressesV2` serialize from the effective normalized address graph; `addressesV2` preserves hydrated connection cursors where captured, honors `reverse` / cursor windowing for normalized local rows, and returns Shopify-like empty connections when no address records exist.
- `CustomerInput.addresses` is modeled on `customerCreate`: submitted rows stage into the same normalized address graph and `Customer.defaultAddress` points at the first input row. On `customerUpdate`, a supplied `input.addresses` list is authoritative: id-bearing rows update existing customer addresses in place, id-less rows create replacement addresses, unknown ids return `Customer address does not exist`, and omitted `addresses` leaves the current address set unchanged. Captured Admin GraphQL 2026-04 evidence covers these id-aware update and replacement semantics; `customerSet(input.addresses)` replacement remains separate from the dedicated address root lifecycle.
- Captured Admin GraphQL 2025-01 evidence for address country/province validation backs `customerCreate(input.addresses)`, `customerAddressCreate`, and `customerAddressUpdate` in the current Rust slice. Unknown country names/codes return `Country is invalid` on the country field path, and province names/codes that do not belong to the resolved country return `Province is invalid` on the province field path. Shopify resolves `countryCode` / `countryCodeV2` before display `country`; when `countryCode: US` is paired with `country: "Canada"`, Shopify accepts the input and normalizes the address to `United States`. Countries without zones such as `SG` ignore submitted province values and return `province` / `provinceCode` as `null`.
- Captured Admin GraphQL 2025-01 evidence for customer address input guardrails backs `customerCreate(input.addresses)`, `customerAddressCreate`, and `customerAddressUpdate` in the current Rust slice. Address strings are stripped before staging, and `address1`, `address2`, `city`, `company`, and `zip` reject values longer than 255 characters. Address strings reject HTML tags and emoji; `city`, `zip`, and address `phone` also reject URL substrings, while address `phone` rejects HTML tags. Dedicated `customerAddressCreate` accepts an all-blank address after normalizing blank strings to `null`; nested `CustomerInput.addresses` rejects an all-blank entry with `Customer address cannot be blank.`.
- Standalone `customerAddressCreate` and `customerAddressUpdate` normalize explicitly submitted address `phone` values before staging: formatted international values carrying a leading `+` become compact E.164, digit-shaped no-`+` values use the effective address country, and unparseable values remain raw. The effective country is resolved from the submitted address country or the existing address country on updates. The resulting value is reflected in the mutation `address`, `Customer.defaultAddress`, and `Customer.addressesV2` read surfaces.
- Captured Admin GraphQL 2025-01 evidence for `CustomerInput.emailMarketingConsent` and `CustomerInput.smsMarketingConsent`, plus 2026-07 evidence for `CustomerInput.whatsAppMarketingConsent`, covers inline consent handling and create-time validation branches. `customerCreate(input: ...)` accepts inline email and SMS marketing consent values and immediately reflects them through `Customer.defaultEmailAddress`, `Customer.defaultPhoneNumber`, `emailMarketingConsent`, and `smsMarketingConsent`; create-time `NOT_SUBSCRIBED` is allowed even though the dedicated update mutations reject it. `customerCreate` rejects email consent without an email (`field: ["emailMarketingConsent"]`), SMS consent without a phone (`field: ["smsMarketingConsent"]`), WhatsApp consent without a phone (`field: ["whatsAppMarketingConsent"]`), and disallowed create-time marketing states such as `REDACTED` with a top-level `INVALID` error and `data.customerCreate: null`; these validation failures do not stage or log a local customer write. `customerUpdate(input: ...)` rejects inline consent fields with payload `userErrors` that instruct callers to use `customerEmailMarketingConsentUpdate`, `customerSmsMarketingConsentUpdate`, or `customerWhatsAppMarketingConsentUpdate`; the rejected update leaves downstream customer reads unchanged. The dedicated consent mutations remain the supported local path for post-create consent changes. The current fixture covers Shopify's default `OTHER` SMS consent collection source; source-location-specific attribution remains deferred until a capture records a non-default branch.
- Captured Admin GraphQL 2025-01 and 2026-04 evidence for customer address lifecycle uses `MailingAddressInput`, payload field `address`, and delete payload field `deletedAddressId`. `customerAddressCreate` with an unknown customer returns payload `userErrors` with `field: ["customerId"]`; update/delete/default roots with an address ID that exists nowhere return top-level `RESOURCE_NOT_FOUND` GraphQL errors with `data.<root>: null`. When the address ID resolves to another customer's address, ordering depends on whether the submitted customer exists: an unknown customer returns the payload `customerId` / `Customer does not exist` userError first, while an existing customer returns the address-scoped `field: ["addressId"]` / `Address does not exist` userError. Update/delete return `address: null` or `deletedAddressId: null`; default-address selection returns either `customer: null` for the missing customer or the unchanged existing customer for the address-scoped error.
- `customerAddressUpdate` also validates a nested `MailingAddressInput.id` before applying updates. When the nested id differs from the `addressId` argument, Shopify returns payload `userErrors` with `field: ["addressId"]` and message `The id of the address does not match the id in the input`; the local handler returns the same public shape and leaves both the target address and the nested-id address unchanged. Explicit `id: null` is captured as the same mismatch, while omitted nested id is treated as absent and a matching nested id updates normally. Public Admin GraphQL exposes these address payload errors as generic `UserError`, so no `code` field is selectable in 2025-01 or 2026-04.
- The customer-address lifecycle parity spec now replays the main captured chain through the generic proxy-vs-recording runner: setup customer, create/update/default/delete address mutations, downstream `customer` / `customerByIdentifier` reads, omitted `setAsDefault`, duplicate-address validation, ordering read, default-address deletion, and final read. Local replay materializes generated IDs between target requests so the chain validates staged read-after-write behavior without runtime Shopify writes. The remaining isolated validation branches stay in `customer-draft-flow`.
- `MailingAddressInput` blank and normalization behavior is fixture-backed for the local slice: `{}` creates a blank address, inherited customer `firstName` / `lastName` populate missing or blank address names, empty address strings normalize to `null`, invalid `countryCode` / country-specific `provinceCode` values return payload userErrors from the generated Shopify Atlas country/zone metadata, and arbitrary postal text is accepted.
- Duplicate `customerAddressCreate` payloads are rejected with `field: ["address"]` and `Address already exists`.
- Address updates recompute derived `formattedArea` from the effective city/province/country fields unless Shopify provided an explicit formatted value, and newly created addresses receive a monotonically increasing local position so post-delete creates do not reorder older addresses. Deleting the current default address promotes the next remaining normalized address to `Customer.defaultAddress`; `setAsDefault: false` and omitted/null `setAsDefault` do not replace an existing default. The capture did not find a maximum-address failure through 105 created addresses, so local staging does not impose a smaller artificial limit.
- Customer payment method reads are modeled from normalized `customerPaymentMethods` state only. The local serializer supports `customerPaymentMethod(id:, showRevoked:)`, `Customer.paymentMethods(showRevoked:)`, and generic `node(id:)` / `nodes(ids:)` for active normalized payment methods. Revoked rows are hidden unless `showRevoked: true` is selected on the dedicated payment-method surfaces, and revoked or unknown payment-method IDs return `null` through generic Node reads. The customer-level connection honors `first` / cursor windowing, `reverse`, aliases, and computed `pageInfo`.
- `CustomerPaymentMethod.instrument` is stored as a selected union payload keyed by `__typename`, so seeded fixtures can serialize credit-card and PayPal billing-agreement fragments without local vaulting. `subscriptionContracts` on the payment method is serialized as a normal connection from seeded link rows; the customer-level `subscriptionContracts` field remains empty/no-data until separately modeled.
- Customer payment method writes stage locally without storing real payment credentials. Credit-card and PayPal create/update store scrubbed instrument shells, remote create stores an incomplete `instrument: null` method, Shop Pay agreement duplication uses a proxy-local non-secret token, Shop Pay update-url returns a non-deliverable `shopify-draft-proxy.local` URL, and revoke marks the normalized method revoked so read filtering observes the state change. Generic Node projection exposes only selected scrubbed fields from that normalized state; cardserver sessions, card digits, gateway tokens, masked numbers, and notification delivery are not stored or fabricated. Non-Shop-Pay duplication/update-url attempts return `INVALID_INSTRUMENT`, same-shop duplication returns `SAME_SHOP`, and blank duplication billing-address fields return `BLANK` userErrors. Live success-path capture remains blocked by missing `read_customer_payment_methods` / `write_customer_payment_methods` scopes and sensitive payment material.
- Store credit accounts are modeled as sensitive balance records. Snapshot mode never creates a store credit account merely because a customer exists: `Customer.storeCreditAccounts` remains an empty connection until normalized account state is seeded or a local mutation updates an existing account, and `storeCreditAccount(id:)` returns `null` for unknown IDs. The customer-level account connection applies the local `query:` slice for account id and balance currency terms before windowing.
- `storeCreditAccountCredit` and `storeCreditAccountDebit` accept `StoreCreditAccount`, `Customer`, and `CompanyLocation` IDs. Account IDs resolve existing normalized accounts; owner IDs resolve the owner's current local account for the input currency, and first credit to a known customer or company location creates a local store credit account without runtime Shopify writes. In LiveHybrid mode, a cold Customer owner or StoreCreditAccount target is read from Shopify through a narrow preflight query, staged locally, and then credited/debited locally; Snapshot mode still returns the local null/empty/missing-owner behavior when the resource is absent. They update the account balance, append local transaction rows with concrete `StoreCreditAccountCreditTransaction` / `StoreCreditAccountDebitTransaction` typenames, and keep credit `remainingAmount` aligned as later debits consume credited value. Direct account reads, nested customer account reads, and generic `node(id:)` / `nodes(ids:)` reads for the account and credit/debit transactions observe the changed balance and transaction money shapes. Local transaction IDs currently use the generic `StoreCreditAccountTransaction` allocator while projecting the concrete transaction `__typename`, so callers should key behavior from the selected typename and fields rather than assuming the GID resource type is concrete. Both roots reject non-positive amounts, currency mismatches, missing accounts/owners, and debit overdraws with Shopify-style GraphQL enum userError codes. Account or owner resolution runs before amount validation, and credit checks `expiresAt` against the current proxy clock before non-positive amount once the account is resolved. Non-positive credit amounts use Shopify's credit-flavored message ("A positive amount must be used to credit a store credit account"), while non-positive debit amounts use the debit-flavored wording ("A positive amount must be used to debit a store credit account"). Credit accepts ordinary public input `CurrencyCode` values captured as live-accepted, including `CHF`; result-only currency codes such as `USDC` and `XXX` return top-level `CURRENCY_CODE_INVALID` errors before staging. Credit also rejects past or present `expiresAt` values and credits that would exceed the captured account balance ceiling with `CREDIT_LIMIT_EXCEEDED` on `["creditInput", "creditAmount", "amount"]`; debit does not expose `UNSUPPORTED_CURRENCY`, so a debit currency that differs from the account continues into account-resolution, currency-mismatch, or insufficient-funds validation.
- Captured Admin GraphQL 2026-04 evidence shows `StoreCreditAccountCreditInput.notify` is a public input field and the local credit path preserves the submitted boolean in the staged transaction/state and original mutation log for commit replay; runtime handling still does not deliver email. The same evidence shows `StoreCreditAccountCreditInput.attribution` is not public in 2026-04, and `StoreCreditAccountDebitInput` exposes neither `notify` nor `attribution`; the proxy therefore rejects those fields with Shopify-style top-level `INVALID_VARIABLE` schema errors instead of staging them.
- Shopify accepts a bare `origin` selection on store-credit transactions even though introspection exposes `StoreCreditAccountTransactionOrigin` as a union. The executor applies a narrow engine-only compatibility selection for this captured exception, preserving the caller-visible `origin: null` response while leaving normal union selection validation intact.
- Store-credit live evidence uses the canonical conformance auth helper against `harry-test-heelo.myshopify.com`: schema introspection captured the account fields (`id`, `balance`, `owner`, `transactions`), transaction fields (`account`, `amount`, `balanceAfterTransaction`, `createdAt`, `event`, `origin`), credit/debit input shapes, and payload fields. The recorder creates a disposable customer, calls `storeCreditAccountCredit` with that customer ID so Shopify creates a real store credit account, captures account-ID validation branches (`ACCOUNT_NOT_FOUND`, `MISMATCHING_CURRENCY`) including the debit unsupported-currency asymmetry, and owner-ID validation branches (`EXPIRES_AT_IN_PAST`, `NEGATIVE_OR_ZERO_AMOUNT`, `INSUFFICIENT_FUNDS`, `CREDIT_LIMIT_EXCEEDED`), confirms combined invalid-input precedence for missing account IDs and past credit expiry, confirms a `CHF` owner-ID credit creates a separate account, records result-only `USDC`/`XXX` credit-input rejection, records public-schema rejection for unsupported attribution/notify fields, records read-only preflight cassettes for cold existing Customer, existing StoreCreditAccount, and missing StoreCreditAccount mutation-first replay, replays account-ID credit/debit mutations plus downstream direct/nested balance reads, debits captured balances back to zero, and deletes the disposable customer. The generic Node evidence creates the same disposable customer/store-credit graph through public mutations and compares `nodes(ids:)` payloads for the customer, mailing address, account, credit transaction, debit transaction, and never-created null entries.
- Captured Admin GraphQL 2025-01 evidence for `customerCreate` and `customerUpdate` now covers the first CustomerInput validation long tail. The local model reproduces payload `userErrors` for invalid email (`["email"]`, `Email is invalid`), invalid phone (`["phone"]`, `Phone is invalid`), duplicate email/phone (`Email has already been taken`, `Phone has already been taken`), invalid locale (`["locale"]`, `Locale is invalid`), oversized tags (`["tags"]`, 255-character limit), oversized first/last name (255-character limit), oversized note (5000-character limit), and create attempts with a caller-supplied `CustomerInput.id` (`["id"]`, `Cannot specify ID on creation`). Failed validations return `customer: null` and do not mutate normalized customer rows or customer-owned metafields.
- Captured Admin GraphQL 2025-01 evidence for customer tag and note limits now covers comma-separated tag splitting inside `tags` array entries, case-insensitive tag de-duplication while preserving the first submitted casing, the resolver-side 250-tag cap after comma splitting, and the 5000-character note cap across `customerCreate`, `customerUpdate`, and `customerSet`. Public Admin API 2025-01 and 2026-04 expose these payload errors as generic `UserError` without a selectable `code` field; the proxy still projects compatibility `TOO_MANY_TAGS` / `TOO_LONG` codes when a local caller selects `code`.
- Captured Admin GraphQL 2025-01 evidence for `customerUpdate` also enforces the required identity precondition after applying the input. Updates that would leave `firstName`, `lastName`, `email`, and `defaultPhoneNumber.phoneNumber` all blank return `customer: null` with `userErrors[{ field: null, message: "A name, phone number, or email address must be present" }]`, and the existing customer remains unchanged on a subsequent `customer(id:)` read. Local staging preserves that no-mutation behavior and emits `code: "INVALID"` when clients select the proxy's compatibility `code` field.
- Captured Admin GraphQL 2025-01 evidence for `customerCreate` enforces the same identity precondition before staging: `firstName`-only and `lastName`-only inputs succeed with `email` / `phone` left `null` and become visible through downstream `customer(id:)` and `customerByIdentifier(identifier: { id })` reads. The capture records `customers(first:, query: "")` readback as supporting evidence; local runtime coverage also verifies staged name-only customers are visible through `customers(query:)`. All-blank input returns `customer: null` with `userErrors[{ field: null, message: "A name, phone number, or email address must be present" }]`; the local proxy emits compatibility `code: "INVALID"` when callers select `code`.
- Captured successful CustomerInput normalization trims blank `firstName`/`lastName`/`phone` to `null`, preserves blank `note` as an empty string, preserves explicit `null` scalar inputs as `null` on update, defaults created-customer `locale` to the shop's primary locale when omitted, and trims, de-duplicates, and lexicographically sorts tags. These normalized values are reflected in mutation payloads and downstream `customer`, `customerByIdentifier`, and `customers` reads.
- `customerUpdate` and `customerSet` preserve existing customer fields not addressed by the submitted input, including hydrated account `state`, `dataSaleOptOut`, default contact/consent fields, customer-owned metafields, and customer order-summary fields.
- Customer phone normalization accepts explicit international `+` input directly. Bare digit-shaped phone input uses submitted address country, existing customer address country, or hydrated/restored shop country context, and returns `Phone is invalid` when no usable country context can resolve the calling code.
- Updating a deleted customer or a customer that was merged away returns the same supported unknown-id branch as other missing customers: `customer: null` plus `userErrors[{ field: ["id"], message: "Customer does not exist" }]`.
- Captured Admin GraphQL 2025-01 evidence for `customerDelete` shows completed/recorded orders block deletion with payload `userErrors[{ field: ["id"], message: "Customer can’t be deleted because they have associated orders" }]` and `deletedCustomerId: null`; `customerDelete.userErrors` uses public `UserError`, so `code` is not selectable on this root. Local staging checks normalized order/customer associations before deleting, keeps the customer readable when blocked, and deliberately does not treat draft orders alone as a delete blocker. When callers select deprecated payload `shop { id }`, the local payload returns the restored or hydrated base shop ID when known and `shop: null` when the proxy has no shop identity.
- Customer order-summary reads have a deliberately narrow order-domain bridge. Freshly staged customers with no orders carry `numberOfOrders: "0"`, `amountSpent: { amount: "0.0", currencyCode: <shop currency> }`, `lastOrder: null`, and an empty `orders` connection. Captured Admin GraphQL 2026-04 evidence for `orderCustomerSet` / `orderCustomerRemove` shows immediate customer reads expose the linked order through `Customer.orders`, then return to an empty order connection after removal, but `numberOfOrders`, `amountSpent`, and `lastOrder` remain at the customer record's existing summary values in the immediate read. The proxy therefore derives `Customer.orders` from normalized effective `OrderRecord.customer` relationships and leaves customer-owned summary scalars on `CustomerRecord` unless they were hydrated from Shopify customer data or initialized for a locally staged no-order customer. The `customer-order-summary-read-effects` parity spec seeds the captured order and replays setup, set, read, remove, and read through strict proxy-vs-recording comparison; expected differences are limited to local customer IDs and opaque order cursors.
- Captured Admin GraphQL 2026-04 evidence for `customerSet` supports a local slice: create without `identifier`, update by `identifier.id`, upsert/update by `identifier.email` or `identifier.phone`, and upsert/update by `identifier.customId` when the custom id points at a valid CUSTOMER metafield definition of type `id` with unique values enabled. The staged input slice is `email`, `firstName`, `lastName`, `locale`, `note`, `phone`, `tags`, `taxExempt`, `taxExemptions`, and address-list replacement when the identifier resolves an existing customer. The executable parity specs now promote the base create/update fixture branches plus captured address replacement, email/phone upsert, custom-id upsert/readback, and validation guardrail branches through the generic proxy-vs-recording runner.
- `customerSet(identifier.id)` validates before create/update routing: an unknown id returns `customer: null` plus `userErrors[{ field: ["input"], code: "NOT_FOUND", message: "Resource matching the identifier was not found." }]` when `code` is selected, and does not fall back to create even when `identifier.email` or `input.email` is also supplied.
- No-identifier `customerSet` creates reject duplicate native contact identifiers locally: existing email returns `field: ["input", "email"]` / `Email has already been taken`, and existing phone returns `field: ["input", "phone"]` / `Phone has already been taken`.
- `identifier.email` and `identifier.phone` require the corresponding input field. Missing values return `The input field corresponding to the identifier is required.` and mismatched values return `The identifier value does not match the value of the corresponding field in the input.` at `field: ["input"]`.
- `customerSet(input.addresses)` is treated as a replacement list for an existing customer: current normalized addresses are removed, input mailing addresses are staged, duplicate replacement rows are coalesced, and `Customer.defaultAddress` follows the first replacement address or `null` for an empty list. A nullable address list (`addresses: null`) is a no-op. Local replay uses stable synthetic address cursors; live Shopify cursors are opaque and are only parity-comparable as expected differences.
- Captured nullable input behavior remains narrow: `taxExempt: null` returns `field: ["input", "taxExempt"]` / `Tax exempt is of unexpected type NilClass`; `taxExemptions: null` is tolerated by the captured branch when `taxExempt` is the only reported validation error.
- `customerSet(identifier.customId)` treats the identifier as a unique customer metafield identity, not as an arbitrary key/value lookup. With a valid CUSTOMER `id` metafield definition, a matching effective customer metafield updates that customer and preserves the identifier metafield; a no-match identifier creates a customer with that metafield. `customerByIdentifier(identifier: { customId })`, `customer(id:)`, aliased `customer.metafield(...)`, and `customer.metafields(...)` read the staged owner-metafield state. Missing definitions, non-`id` definitions, and definitions without enabled unique values return Shopify's top-level `NOT_FOUND` custom-id error with `data.customerSet: null`.
- Admin GraphQL 2026-04 does not expose `CustomerSetInput.metafields`; submitted `input.metafields` and malformed `identifier.customId` variables are rejected as top-level `INVALID_VARIABLE` schema errors before the local resolver runs. Shopify also rejects explicit `capabilities.uniqueValues.enabled: false` on `type: "id"` metafield definitions with `CAPABILITY_REQUIRED_BUT_DISABLED`; omitting that capability for `type: "id"` auto-enables unique values.
- Unsupported `customerSet` input or identifier fields that are outside the captured public input schema should fail through schema validation rather than falling into local staging.
- Staged `customerMerge` chooses the resulting customer from normalized customer state using Shopify's captured selection order: a valid `overrideFields.customerIdOfEmailToKeep` wins first, then exactly-one-email wins, then both-email consent/account-state tiebreaks (`SUBSCRIBED`/`PENDING`, then `ENABLED`/`INVITED`) apply, and ties or neither-email inputs default to `customerTwoId`.
- The selected `customerMerge` result row is updated locally, the other customer is marked deleted, the source-to-result redirect is recorded in `mergedCustomerIds`, and the observed merge job/result shape is recorded in `customerMergeRequests`. The `customer-merge-selection-rules` parity spec captures override customerOne, only-customerOne-has-email, account-state customerOne, and neither-has-email customerTwo-default branches.
- `customerMergePreview` and `customerMergeJobStatus` resolve from normalized customer/merge-request state. The first local merge slice supports customers already present in staged state or hydrated base state and does not fetch unknown customer ids during the supported mutation path.
- Customer merge job ids recorded in `customerMergeRequests` resolve through both `job(id:)` and `node(id:)` as completed local `Job` payloads after staging. Unknown arbitrary Job ids still follow the admin-platform no-data behavior for `node(id:)`.
- `customerMerge` validates required IDs before handler execution: omitted required IDs return Shopify's top-level `missingRequiredArguments` envelope, while blank literal ID strings return top-level `argumentLiteralsIncompatible` coercion errors for each blank ID. Payload validation checks `customerOneId` existence, then `customerTwoId` existence, then same-id rejection. A duplicated nonexistent customer ID therefore returns `field: ["customerOneId"]` / `INVALID_CUSTOMER_ID`; a duplicated existing customer ID returns `field: null` / `Customers IDs should not match`. The base customer merge parity fixture captures these branches.
- `customerMerge` also runs synchronous blocker detection before staging a merge. Captured 2025-01 evidence covers combined tags over 250, combined notes over 5,000 characters, and an assigned gift card; each returns `resultingCustomerId: null`, `job: null`, and immediate `userErrors` with `code: INVALID_CUSTOMER` instead of enqueueing a job. Local runtime coverage also blocks customers with normalized payment methods that carry subscription contracts; the public conformance workflow does not currently expose a reproducible subscription-contract setup path. Public Admin GraphQL exposes these mutation blockers through `customerMerge.userErrors`; the proxy additionally projects the same local blocker rows through compatibility `customerMergeErrors` selections, including `field`, `code`, and `block_type`.
- The attached-resource customer-merge capture is a dedicated scenario separate from the base customer merge parity fixture. It extends local `customerMerge` staging for normalized customer-owned addresses, customer-owned metafields, and normalized orders. Shopify selected `customerTwoId` as the resulting customer, kept the result customer's default address, appended the source customer's address to `addressesV2`, preserved result-side metafield conflicts, copied source-only metafields under a new metafield id, and moved the source order to the result customer with the result customer's email. `numberOfOrders` and `lastOrder` stayed at the result customer's existing summary values in the captured immediate post-completion reads.
- Polling evidence observed `customerMergeJobStatus` as `IN_PROGRESS` immediately after the mutation and `COMPLETED` on the next poll. The local merge request is still recorded as completed immediately after staging because the local mutation has no asynchronous worker, but the fixture keeps the live polling samples for future asynchronous fidelity work.
- Captured Admin GraphQL 2025-01 evidence for `customerAddTaxExemptions`, `customerRemoveTaxExemptions`, and `customerReplaceTaxExemptions` stages against `Customer.taxExemptions` only; it does not flip the separate `taxExempt` boolean. Unknown customers return payload `userErrors` at `["customerId"]` with `Customer does not exist.` Invalid enum variables are top-level `INVALID_VARIABLE` GraphQL errors before payload execution.

- The `dataSaleOptOut` root remains documented under the privacy endpoint group, but its local read-after-write state is stored on `CustomerRecord`. Existing-email opt-out flips `Customer.dataSaleOptOut` to `true`; unknown valid emails create a local opted-out customer; invalid email strings return the captured `FAILED` userError shape.
- `customerAccountPage(id:)` and `customerAccountPages` are read-only Customer Account system-page roots. Snapshot mode resolves these roots only from normalized local account-page state and otherwise returns `null` / empty connections instead of inventing pages. - `customerRequestDataErasure` and `customerCancelDataErasure` are treated as sensitive customer privacy side effects. The `customer-data-erasure-request-cancel` parity spec replays the success and unknown-customer mutation payloads locally while runtime tests cover no-upstream staging, meta audit state, and replay order.

### Outbound email and activation buffering

The customer surface includes roots that look related but have different safety profiles:

- `customerEmailMarketingConsentUpdate` and `customerSmsMarketingConsentUpdate` are customer state changes, not outbound notification sends. They are safe to stage locally once captured behavior is modeled because downstream reads can observe the changed consent fields.
- `customerGenerateAccountActivationUrl` returns a sensitive, expiring customer-facing link. Runtime support only succeeds for `DISABLED` or `INVITED` customers, records a stable synthetic account-activation token on the local customer row, synthesizes a non-deliverable token-bearing activation URL, and keeps the original raw mutation for commit replay instead of asking Shopify for a live URL. Other customer states return `account_already_enabled`.
- Storefront customer-auth roots share this customer Store. A locally staged Admin `customerCreate` starts as `DISABLED`, can be activated by the Storefront `customerActivate` / `customerActivateByUrl` roots using the locally generated activation URL, and then reads back as `ENABLED` through later Admin `customerUpdate` payloads and Storefront token-authenticated `customer` queries. Admin `customerUpdate` preserves the existing account state unless the submitted mutation explicitly changes modeled customer-account state, and Admin `customerDelete` tombstones the shared row so existing Storefront access tokens no longer resolve a customer.
- `customerSendAccountInviteEmail` is an explicit outbound email side effect. Runtime support buffers the original mutation locally and does not deliver email at runtime; delivery happens only if the staged mutation log is committed to Shopify.
- `customerSendAccountInviteEmail` only stages the invite for locally invitable account states. `DISABLED` customers transition to `INVITED`; customers already in `ENABLED` or another ineligible state return `ACCOUNT_ALREADY_ENABLED` on `customerId` and keep their current state.
- `customerPaymentMethodSendUpdateEmail` remains unsupported as a primary mutation root until customer-payment ownership and email buffering have their own captured implementation path.
- `customerPaymentMethodGetUpdateUrl` synthesizes a non-deliverable local URL only for Shop Pay agreement instruments. Do not replace it with a live Shopify URL fetch during supported runtime handling.

Do not mark outbound email roots implemented by proxying them upstream. Support means local validation/buffering plus original raw mutation retention for commit-time replay; it does not require pretending the runtime email or URL side effect already happened in Shopify.
