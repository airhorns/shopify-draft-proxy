The products group is product-first and deep, with registry entries for supported local behavior and explicit unsupported gaps. It covers product roots plus directly related inventory, metafield, collection, publication, tag, helper, feedback, and product-adjacent media boundaries.

### Behavior notes

- Product feed reads support Shopify-like no-data behavior in snapshot mode and read-after-write behavior for locally staged feeds. Captured 2025-01 `harry-test-heelo` evidence returns `productFeed(id:)` as `null` for an absent feed id and `productFeeds(first:)` as an empty connection with empty `nodes`/`edges`, `hasNextPage: false`, `hasPreviousPage: false`, and null cursors. Locally staged `productFeedCreate` validates `country` against `CountryCode`, validates `language` against `LanguageCode`, returns `INVALID` user errors for unknown values, and returns `TAKEN` for a duplicate `(country, language)` pair before staging. Successful creates return an `ACTIVE` `ProductFeed` with deterministic id `gid://shopify/ProductFeed/{country}-{language}`; the same effective feed is also readable through generic `node(id:)` / `nodes(ids:)` dispatch. `productFullSync` exposes only `id` and `userErrors`: selecting `job` returns Shopify's top-level undefined-field error for `ProductFullSyncPayload`, absent feeds return `field: ["id"]` with `code: null`, invalid `updatedAtSince` / `beforeUpdatedAt` ranges return `field: ["updatedAtSince"]`, and any staged feed id can sync to a synthetic operation id without staging a `Job`. `productFeedDelete` removes staged feeds so downstream `productFeed(id:)`, `productFeeds(...)`, and generic Node reads return null/empty again. Supported feed mutations do not proxy upstream at runtime. The same live store still returns a top-level `NOT_FOUND`, `Unable to find channel for product feed`, for Shopify-side `productFeedCreate(country: US, language: EN)`, so success conformance is documented as channel-configuration-blocked for that shop. Runtime tests cover local product-full-sync success and validation; captured 2025-01 evidence backs the public payload fields and unknown-feed null-code branch.
- Selling-plan group roots are tracked in the registry and have checked-in capture evidence, but they are not current Rust local dispatch roots. The proxy does not claim supported selling-plan lifecycle behavior or downstream product/variant membership read-after-write effects until a store-backed model exists.
- Product bundle mutations are handled locally without runtime Shopify writes. Successful `productBundleCreate` / `productBundleUpdate` invocations return a `ProductBundleOperation` in the mutation payload with initial `status: CREATED`; the local operation then completes immediately for readback. Successful `productBundleCreate` stages a synthetic bundle product and exposes it through `productOperation(id:)` and `node(id:)` as `status: COMPLETE` with `product`. Successful `productBundleUpdate` links the completed operation to the updated product. Immediate validation failures preserve Shopify's captured 2025-01 shape: root `userErrors` are returned, `productBundleOperation` is `null`, and no bundle operation is staged. Captured guardrails cover empty components, unknown update products, missing component products, invalid option mappings, component quantity above 2000, and `quantityOption` values with fewer than two entries. The same fixture confirms `quantity: 0`, blank `quantityOption.name`, and zero quantity option values are accepted when the component option mapping is otherwise valid. The configured 2025-01 live schema rejects `ProductBundleCreateInput.consolidatedOptions` before resolver execution, so local consolidated-option validation is runtime-test-backed rather than live parity-backed for this version.
- `productVariantRelationshipBulkUpdate` stages normalized `ProductVariantComponent` rows keyed by parent variant. Downstream `ProductVariant.requiresComponents` and `ProductVariant.productVariantComponents(...)` resolve from the effective local relationship graph. Captured validation parity covers unknown parent/component variants returning `field: ["input"]`, code `PRODUCT_VARIANTS_NOT_FOUND`, and parent-as-child input returning `field: ["input"]`, code `CIRCULAR_REFERENCE`. Local validation rejects parent rows that specify both `parentProductId` and `parentProductVariantId`, duplicate parent rows, duplicate component IDs within one create set, component quantities outside `1..9999`, and update/remove rows whose component variant is not currently a child of the parent; rejected inputs return user errors without staging the mutation.
- `combinedListingUpdate` stages parent/child combined-listing relationships locally. The parent product receives `combinedListingRole: PARENT`, child products receive `combinedListingRole: CHILD`, and downstream `Product.combinedListing.combinedListingChildren(...)` resolves child products plus the matched parent variant from submitted parent option values. Captured validation parity covers unknown parent products returning code `PARENT_PRODUCT_NOT_FOUND`; the local mutation also rejects non-parent products, missing `optionsAndValues` for add/edit requests, parent-as-child inputs, duplicate child ids, unknown child products, empty selected parent option values, overlong titles, already-staged child products, and edit/remove overlap with the captured public Admin API error codes.
- Product-domain metafields are normalized as owner-scoped records for `Product`, `ProductVariant`, and `Collection` owners. Besides `id`, `namespace`, `key`, `type`, and `value`, hydrated and staged records carry `compareDigest`, `jsonValue`, `createdAt`, `updatedAt`, and `ownerType` for owner-scoped parity. Generic `node(id:)` / `nodes(ids:)` resolves effective product-domain `Metafield` GIDs through the same owner-scoped records; Metafield `definition` still serializes as `null` for product metafield node selections until a product-owned fixture returns nested definition data. Definition records themselves are modeled through the metafields endpoint group.
- Local `metafieldsSet` support covers product, product variant, and collection owners only. It validates the full input batch before replacing each affected owner metafield set, supports compare-and-set through `compareDigest`, treats `compareDigest: null` as a create-only guard, and preserves Shopify-like atomic no-write behavior when any modeled resolver error is returned. For matching staged/effective metafield definitions, it infers omitted input `type`, rejects explicit type mismatches, and applies the fixture-backed `max` and `regex` validations represented in the local definition model. Customer, order, draft-order, shop, discount, and other owner families remain scoped to their own endpoint groups or future issues.
- `metafieldsDelete` uses the same product-domain owner scope and returns ordered `deletedMetafields` entries, including `null` for missing namespace/key rows. Downstream `product(id:)`, `productVariant(id:)`, and `collection(id:)` reads expose staged owner-specific singular `metafield(namespace:, key:)` and `metafields` connection results without live writes. Captured parity includes an ordinary operation-name product-owner set/read/delete/read flow so the generic path is not tied to conformance fixture operation names.
- Product search uses the shared Shopify-style search parser. Endpoint-specific product behavior includes boolean grouping, quoted values, field comparators, simple term-list searches, variant search terms, sort keys, and captured connection cursor/pageInfo baselines.
- In Rust LiveHybrid parity, cold `product`, `productByIdentifier`, `products`, and `productsCount` reads use Pattern 1 cassette passthrough when no relevant local product state exists. Product-owned `metafield` / `metafields` selections stay local until the product-metafields parity slice is cassette-migrated. Once staged products, product deletes, or product-owned metafield shells are present, those same roots stay on the local serializer so read-after-write and owner-scoped metafield behavior remains local without runtime Shopify writes.
- `productCreate`, `productUpdate`, and `productSet` stage `category` and `requiresSellingPlan` from the public 2025-01 product input shapes without runtime Shopify writes. `category` is stored in the product category slot and is exposed through downstream `Product.category { id fullName }` reads; the local input reader also understands older wrapper objects with `productTaxonomyNodeId` when the request shape reaches the handler. Live 2025-01 evidence shows `Product.productCategory` and the deprecated category wrapper inputs are not exposed by the public schema, so those wrappers remain compatibility handling rather than public parity targets for this version. Explicit `requiresSellingPlan` input values take precedence over the default `false`; selling-plan group membership does not currently derive this field because selling-plan lifecycle roots are unimplemented.
- `productCreate.collectionsToJoin` reuses the collection membership staging path used by collection add-products mutations. Joined custom collections are immediately visible through both `Product.collections` and `Collection.products` / `hasProduct` / `productsCount` reads. Public 2025-01 capture showed unknown collection GIDs in `collectionsToJoin` are ignored with empty `userErrors`, so the local handler skips unknown collection IDs rather than rejecting them; smart collections are still rejected locally because manual membership writes are not valid for smart collections.
- Captured productCreate input-field parity lives in `productCreate-category-parity`, `productCreate-requires-selling-plan-parity`, and `productCreate-collections-to-join-parity`. The same recorder stores validation probes showing invalid category GIDs are top-level `INVALID_VARIABLE`, unknown taxonomy category GIDs are top-level `INVALID_PRODUCT_TAXONOMY_NODE_ID`, `category + productType` is accepted by public 2025-01, and unknown `collectionsToJoin` IDs are ignored.
- `productCreate(product.status)` and `productChangeStatus(status:)` validate against the effective public `ProductStatus` enum before staging. Invalid literals return Shopify's top-level `argumentLiteralsIncompatible` error shape; invalid variables return `INVALID_VARIABLE` coercion errors, and neither branch writes to the local mutation log. Public 2025-01 accepts `ACTIVE`, `ARCHIVED`, and `DRAFT`; `UNLISTED` is only accepted on versioned routes where the public schema exposes it, and `SUSPENDED` remains internal-only.
- Live schema introspection confirms product reorder roots for variant order (`productVariantsBulkReorder`), media order (`productReorderMedia`), option/value order (`productOptionsReorder`), and collection product order (`collectionReorderProducts`). Local support covers variant, option/value, and collection product reorder roots with ordered normalized variants, options, option values, and collection membership rows. `productReorderMedia` remains unsupported until product media has a store-backed lifecycle model.
- `productVariantsBulkReorder` stages an ordered effective variant list from `ProductVariantPositionInput.position` and exposes that ordering through downstream `product.variants(...)` and `productVariant(id:)` reads without runtime Shopify writes.
- `productOptionsReorder` stages the submitted option order and optional option-value order. It remaps variant `selectedOptions`, derives variant titles from the reordered selected-option tuple, and sorts downstream `product.variants(...)` by the reordered option/value sequence. Captured 2025-01 parity covers typed payload validation for unknown option ids/names, unknown option value ids/names, duplicate option/value names, and mixed id/name selectors. Public Admin GraphQL rejects missing oneOf selectors and the internal `position` key at schema validation time before returning `ProductOptionsReorderPayload`; the proxy's inline parser still rejects those shapes locally with the corresponding reorder guardrail codes.
- `productByIdentifier` supports `identifier.id` and `identifier.handle` against effective local product state. `identifier.customId` resolves product metafields backed by an effective PRODUCT metafield definition whose type is `id`; without that definition, snapshot/local reads return Shopify's captured top-level `NOT_FOUND` error, `Metafield definition of type 'id' is required when using custom ids.`
- `productVariantByIdentifier` supports `identifier.id` against effective local variant state. `identifier.customId` returns `null` until unique variant metafield identifier evidence and indexing are modeled.
- Top-level `productVariants` and `productVariantsCount` resolve from effective local product variants. Supported local query terms are `id`, `product_id`, `title`, `sku`, `barcode`, `vendor`, `product_type`, and `tag`, with the shared Shopify-style boolean parser. Local sorting supports the common `ID`, `TITLE`, `SKU`, `POSITION`, and `INVENTORY_QUANTITY` paths, with connection serialization handled by the Rust connection helpers in `src/proxy.rs`.
- `productTags`, `productTypes`, and `productVendors` serialize distinct sorted `StringConnection` values from effective local products. Empty snapshot state returns empty `nodes`/`edges` with false `pageInfo` booleans and null cursors.
- Product tag normalization is fixture-backed by the 2025-01 `productUpdate-tag-normalization` and `tags-normalization-live-parity` captures. `productCreate`, `productUpdate`, `productSet`, `tagsAdd`, and `tagsRemove` trim tag edges, split comma-separated tag strings for tag mutations, deduplicate and remove tags by a case-insensitive identity, and sort returned `Product.tags` by that identity while preserving the first surviving tag's casing and internal whitespace. Shopify returned a top-level `MAX_INPUT_SIZE_EXCEEDED` error for 251 submitted product tags, and a payload `Product tags is invalid` userError for a tag longer than 255 characters; the local handlers mirror those branches without runtime Shopify writes.
- The deprecated `productCreate(input: ProductInput!)` shape remains accepted by public Admin GraphQL 2026-04 even though introspection lists only `productCreate(product:)`. Captured `harry-test-heelo` evidence shows `input.id` returns a payload `userErrors[{ field: ["input"], message: "id cannot be specified during creation" }]` before scalar validation and without staging a product. The same capture shows `input.variants` is rejected before resolver execution because `ProductInput` does not define `variants`; local schema validation mirrors that top-level `argumentNotAccepted` / `INVALID_VARIABLE` branch instead of accepting ignored variant keys.
- Generic `tagsAdd` / `tagsRemove` dispatch by taggable GID type for the modeled owner families: `Product`, `Order`, `DraftOrder`, `Customer`, and `Article`. Successful non-product updates stage into the owning domain store and downstream `order(id:)`, `draftOrder(id:)`, `customer(id:)`, and `article(id:)` reads expose the updated tags without runtime Shopify writes. Captured non-product tag mutations split single comma-separated strings and comma-containing list entries, strip blanks, deduplicate additions by case-insensitive identity, and sort returned tag arrays by case-insensitive identity. Customer `tagsAdd` additionally lowercases newly added all-uppercase variants in the captured path, and customer removes are case-insensitive. Order, Article, and DraftOrder captures showed case-insensitive add dedupe/sort but resource-specific remove behavior: removing exact `Red` succeeds, while removing the case variant `red` leaves an existing `Red` tag in place. `DiscountNode` remains unsupported until local discount tag state and the Shopify `discount_tags_api` flag behavior are modeled. GIDs outside the supported taggable set return Shopify's captured top-level `invalid id` / `RESOURCE_NOT_FOUND` shape instead of a product-scoped payload userError.
- The executable 2025-01 live captures for `productChangeStatus`, `tagsAdd`, and `tagsRemove` against `harry-test-heelo.myshopify.com` compare mutation payloads plus immediate downstream product/status/tag-filtered reads. Status and tag search are modeled as eventually consistent indexes: direct `product.status` / `product.tags` changes are visible immediately, status-filter and tag-filter catalog searches can keep using the pre-mutation indexed values during the lag window, and returned product nodes still serialize the current product fields.
- `productSavedSearches` is handled by the saved-search domain model. It returns staged product saved searches with downstream read-after-write behavior and otherwise preserves the captured empty connection shape for product saved searches.
- `productSet` runs local shape guardrails before staging. Captured 2025-01 Shopify evidence returns top-level `MAX_INPUT_SIZE_EXCEEDED` errors, without a `data` envelope, when `input.variants` has more than 2048 entries or a variant's `inventoryQuantities` has more than 250 entries. The local handler mirrors those top-level branches, rejects product option counts above 3, option values above 100 per option, and files above 250 with payload `INVALID_INPUT` userErrors, and returns `PRODUCT_DOES_NOT_EXIST` on the referenced input field when `input.id` / `input.productId` resolves to nothing. Captured 2026-04 evidence for the `ResourceSetIdentifierExtension` guardrail rejects `identifier:` plus `input.id` before staging with `ID_NOT_ALLOWED`, field `["input"]`, and message `The id field is not allowed if identifier is provided.` Suspended effective products are rejected locally with `INVALID_PRODUCT` on `["input"]`; public conformance setup cannot create suspended products, so that branch is covered by focused Rust runtime tests rather than a live capture.
- `productSet(input:, synchronous: false)` records a local `ProductSetOperation`: the mutation returns a `CREATED` operation with `product: null` and empty `userErrors`, while `productOperation(id:)` can immediately read back the completed operation with `status: COMPLETE`, `product`, and `userErrors`. Unknown operation IDs return `null`, matching the captured helper-root no-data behavior.
- `productDuplicate(productId:, newTitle:, newStatus:)` validates `newStatus` against the public `ProductStatus` enum before staging. When omitted, the duplicate inherits the source product status; when present, the duplicate uses the requested status. Captured parity covers an ACTIVE source with omitted `newStatus` and a DRAFT source duplicated with `newStatus: ACTIVE`, including immediate downstream `product(id:)` status readback.
- `productDuplicate(productId:, newTitle:, synchronous: false)` stages the duplicate product locally, returns the captured async mutation shape with `newProduct: null`, a `ProductDuplicateOperation` in `CREATED` status, and empty mutation-level `userErrors`, then exposes the completed operation through `productOperation(id:)` with `status: COMPLETE`, the source `product`, `newProduct`, and operation `userErrors`. A missing source product follows the 2025-01 `harry-test-heelo` capture: the mutation still returns a created operation with no immediate user errors, while the completed operation carries `userErrors[{ field: ["productId"], message: "Product does not exist" }]` and no products. Local completion is immediate on the next helper read rather than time-delayed, and no supported runtime duplicate path writes upstream.
- `productDelete(input:, synchronous: false)` stages a `ProductDeleteOperation` without deleting the product immediately. The mutation returns `deletedProductId: null`, a `CREATED` operation, and empty `userErrors`; immediate product reads still return the product. A second pending async delete for the same product returns `productDeleteOperation: null` with the captured public userError shape `{ field: null, message: "Another operation already in progress. Please wait until current one is finished." }`. The local helper read exposes the operation as completed with `deletedProductId`, and no supported runtime delete path writes upstream.
- `productDuplicateJob(id:)` models the captured unknown-job read shape as `{ id, done: true }`. Current async `productDuplicate` evidence uses `ProductDuplicateOperation` through `productOperation(id:)`; the job root remains a compatibility helper for the older captured `ProductDuplicateJob` shape.
- `productResourceFeedback(id:)` returns `null` for absent local feedback and returns locally staged feedback after `bulkProductResourceFeedbackCreate`. Product feedback records are keyed by product id and expose `productId`, `state`, `messages`, `feedbackGeneratedAt`, and `productUpdatedAt` without runtime Shopify writes.
- `bulkProductResourceFeedbackCreate` stages `ProductResourceFeedback` records locally for effective products and returns `Product does not exist` userErrors for missing products without staging partial feedback. `shopResourceFeedbackCreate` stages shop-level `AppFeedback` in meta state and returns the mutation payload locally. Live probes against `harry-test-heelo` returned top-level `ACCESS_DENIED` for both feedback mutations because the app lacks `write_resource_feedbacks` and Storefront API / Sales Channel configuration, so live success parity is blocked by app configuration rather than local runtime behavior. Executable local-runtime parity covers product feedback success, missing-product validation, downstream productResourceFeedback reads, and shop feedback staging.
- Feedback mutations validate the schema-backed `ResourceFeedbackState` enum, require at least one message for `REQUIRES_ACTION`, reject messages over 100 characters, reject invalid or future `feedbackGeneratedAt` values, and cap bulk product feedback batches at 50 entries with indexed `["feedback", i, ...]` fields. These validation guardrails are covered by executable local-runtime parity because live resolver capture is blocked by the same resource-feedback app configuration constraints.
- Collection records carry aggregate publication target ids alongside product publication ids. A staged `collectionCreate` starts unpublished; collection publication counts and `publishedOnPublication(publicationId:)` remain unpublished until a local publish mutation adds a target.
- `publishedOnCurrentPublication` is not inferred from aggregate collection publication count. Captured Online Store publishable writes leave it false when the app current publication is not the target.
- Local `publishablePublish` and `publishableUnpublish` currently stage Product and Collection publishables. Product-scoped generic publishable payloads also serialize the selected `shop.publicationCount` from the effective local publication catalog, so explicit publication targets can add derived catalog rows when no publication fixture was seeded. The internal current-channel placeholder used by `publishablePublishToCurrentChannel` updates product aggregate/current-publication state but does not become a shop-level publication row; captured parity shows Shopify keeps `shop.publicationCount` tied to the shop publication catalog. Broader publishable implementers remain unsupported in their own groups.
- Shopify's current docs mark `productPublish` as deprecated in favor of `publishablePublish`; the local product-specific roots remain supported because captured parity fixtures still exercise the deprecated product payload shape used by existing apps. `PublicationInput.publishDate` is accepted as a target field but only online-store channels support future publishing in Shopify, and the field has no effect for unpublish mutations. Subscription-only products (`requiresSellingPlan: true`) are publication-restricted to online-store targets in Shopify; the local aggregate model should not generalize non-online-store success without additional live evidence.
- Top-level `publication(id:)`, `publications(...)`, `publicationsCount(...)`, `publishedProductsCount(publicationId:)`, and deprecated `channel` / `channels` roots resolve from the normalized publication catalog in snapshot/local overlay paths. Empty snapshot state returns `publication: null`, `channel: null`, empty channel/publication connections, and exact zero counts without upstream access. The existing live publication catalog fixture captures non-empty `publications` id/name/cursor shape; runtime tests cover the remaining root family and lifecycle flow.
- `publicationCreate`, `publicationUpdate`, and `publicationDelete` stage normalized Publication rows locally and never perform runtime Shopify writes. `publicationCreate` follows the public `PublicationCreateInput` shape (`catalogId`, `defaultState`, `autoPublish`): omitted `catalogId` creates a local publication, unknown catalogs return `CATALOG_NOT_FOUND`, and unsupported input fields are rejected by schema-style validation before staging. `publicationUpdate` follows the public `PublicationUpdateInput` shape (`publishablesToAdd`, `publishablesToRemove`, `autoPublish`): it adds/removes known Product and ProductVariant publishable IDs from the local publication-membership map, rejects batches over 50, rejects absent or unsupported publishable IDs, and rejects requests that mix products and variants. Deleting a created local publication strips that target from locally modeled publication membership so downstream `publishedOnPublication`, publication counts, and `published_status` filters stop seeing the removed target; the delete payload exposes only `deletedId` and `userErrors`, while the default Online Store publication remains protected from local deletion.
- Product handle generation and validation follows the captured product mutation slice: duplicate title-generated handles are de-duplicated with Shopify's incrementing numeric suffix pattern (`handle`, `handle-1`, `handle-2`, ...), trailing numeric suffixes are stripped only after the original generated candidate is already taken, explicit handles are normalized before uniqueness checks, Unicode letters/numbers are preserved, punctuation-only explicit handles fall back into the `product` handle family, explicit collisions return `['input', 'handle']` userErrors, and explicit handles longer than 255 characters return `['handle']` userErrors without staging partial state. The live reserved-word probe found no product reserved-word rejection for handles such as `admin`, `products`, `collections`, `cart`, `checkout`, or `new`.
- Product option lifecycle staging is fixture-backed for `productOptionsCreate`, `productOptionUpdate`, and `productOptionsDelete`. The current conformance fixtures cover replacing Shopify's default `Title` option with created options, keeping non-variant option values in `optionValues` but out of `values`, renaming and repositioning options, adding/updating/deleting option values, reordering variant `selectedOptions` after option repositioning, and restoring Shopify's default option/variant graph when all custom options are deleted. Local `productOptionsCreate(variantStrategy: CREATE)` also stages variants for the effective option-value Cartesian product, which keeps `values`, `optionValues.hasVariants`, and downstream `product.variants(...)` reads aligned with Shopify's documented CREATE strategy behavior. Captured 2025-01 `harry-test-heelo` evidence for a second CREATE option shows Shopify keeps the existing variants first after filling the new option's first value, then appends the remaining fanout combinations. That shop accepted 110 variants with no `productOptionsCreate.userErrors` (`field`, `message`, and `code` were all absent because the array was empty), which is above Shopify's historical default 100-variant limit and indicates Extended Variants-relevant behavior on this test shop rather than a default-limit rejection. Expected parity differences are limited to generated `ProductOption`, `ProductOptionValue`, and locally generated variant/inventory-item GIDs. Generic `node(id:)` / `nodes(ids:)` support now resolves ProductOption and ProductOptionValue GIDs from the same effective option graph; missing option/value IDs and unsupported adjacent product Node families still return `null` through generic Node dispatch.
- `productOptionsCreate` also retains product-option linked metafield metadata when callers provide `linkedMetafield { namespace key values }` or per-value `linkedMetafieldValue`. When the linked metafield resolves to an effective PRODUCT metafield definition with a `metaobject_definition_id` validation, the proxy records the associated metaobject definition as linked to the product option so later metaobjectDefinitionUpdate guards can reject display-name-key changes locally. Option value display names are derived from effective local metaobject display names when linked values reference locally staged or hydrated metaobjects; otherwise the linked ID remains the conservative fallback.
- `productOptionsCreate` now runs local Shopify-like validation before staging and records rejected branches as failed local mutation-log entries. Captured 2025-01 `harry-test-heelo` evidence covers exact duplicate option names (`DUPLICATED_OPTION_NAME` at `["options","1"]`), empty `values` arrays (`OPTION_VALUES_MISSING` at `["options","0"]`), option names over 255 characters (`OPTION_NAME_TOO_LONG`), existing option-name collisions (`OPTION_ALREADY_EXISTS`), more than three effective product options (`OPTIONS_OVER_LIMIT`), and `variantStrategy: CREATE` fanout beyond 2048 variants (`TOO_MANY_VARIANTS_CREATED`). The same local validator also covers value-name length, duplicate value input, and missing-name guardrails from the Shopify error-code surface in focused Rust tests.
- Product create/set option autogeneration is intentionally asymmetric. Captured `productCreate(productOptions:)` evidence shows Shopify creates one initial variant from the first value of each submitted option, keeps extra submitted values in `optionValues`, and leaves those extra values out of `values` with `hasVariants: false`; it does not create the full Cartesian variant matrix. Captured `productSet(input.productOptions)` evidence shows Shopify requires `input.variants` when product options are supplied, returning `product: null` and `userErrors[{ field: ["input","variants"], message: "Variants input is required when updating product options" }]` instead of implicitly generating combinations. Local staging mirrors both branches without runtime Shopify writes.
- Explicit `productOptionsCreate(variantStrategy: LEAVE_AS_IS)` and an explicit null `variantStrategy` both matched Shopify's default branch in the 2025-01 capture: the standalone existing variant was updated to the first created option value, additional option values remained in `optionValues` with `hasVariants: false`, `values` included only the variant-backed value, and `userErrors` was empty.

### Unsupported and boundary behavior

- Product-domain support is limited to the roots and owner families listed above. Customer, order, draft-order, shop, discount, and other metafield owners remain scoped to their own endpoint groups or future conformance work.

- Product-owned media mutation roots (`productCreateMedia`, `productUpdateMedia`, `productDeleteMedia`, `productReorderMedia`, `productVariantAppendMedia`, and `productVariantDetachMedia`) remain unsupported in the current local dispatch inventory. - Unsupported product-adjacent roots should remain on the generic unsupported path unless a local lifecycle model, downstream reads, and executable evidence exist.
