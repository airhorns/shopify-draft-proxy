### Behavior notes

- Product feed reads support Shopify-like no-data behavior in snapshot mode and read-after-write behavior for locally staged feeds. Snapshot no-data coverage returns `productFeed(id:)` as `null` for an absent feed id and `productFeeds(first:)` as an empty connection with empty `nodes`/`edges`, `hasNextPage: false`, `hasPreviousPage: false`, and null cursors. Locally staged `productFeedCreate` validates `country` against `CountryCode`, validates `language` against `LanguageCode`, returns `INVALID` user errors for unknown values, and returns `TAKEN` for a duplicate `(country, language)` pair before staging. Successful creates return an `ACTIVE` `ProductFeed` with deterministic id `gid://shopify/ProductFeed/{country}-{language}`; the same effective feed is also readable through generic `node(id:)` / `nodes(ids:)` dispatch. `productFullSync` returns the feed id, a pollable `Job`, and empty `userErrors` for a staged feed; absent feeds return `id: null`, `job: null`, `field: ["id"]`, and `code: null`. Invalid `updatedAtSince` / `beforeUpdatedAt` ranges return `field: ["updatedAtSince"]` without staging. `productFeedDelete` removes staged feeds so downstream `productFeed(id:)`, `productFeeds(...)`, and generic Node reads return null/empty again. Supported feed mutations do not proxy upstream at runtime. The same live store returns a payload userError for Shopify-side `productFeedCreate(country: US, language: EN)` because feeds are managed automatically for the channel/spec and required product-feed webhooks are not registered, so success conformance is documented as store/app-configuration-blocked for that shop. Runtime tests cover local product-feed create, product-full-sync success, validation, public payload fields, and the unknown-feed branch.
- Selling-plan group roots are current Rust local dispatch roots documented in the selling-plans endpoint group. Local support is scoped to staged selling-plan groups and locally known product/variant resources; arbitrary live-store selling-plan groups are not hydrated into product reads.
- Product bundle mutations are handled locally without runtime Shopify writes. Successful `productBundleCreate` / `productBundleUpdate` invocations return a `ProductBundleOperation` in the mutation payload with initial `status: CREATED`; the local operation then completes immediately for readback. Successful `productBundleCreate` stages a synthetic bundle product and exposes it through `productOperation(id:)` and `node(id:)` as `status: COMPLETE` with `product`. Successful `productBundleUpdate` links the completed operation to the updated product. Immediate validation failures preserve Shopify's captured 2025-01 shape: root `userErrors` are returned, `productBundleOperation` is `null`, and no bundle operation is staged. Captured guardrails cover empty components, unknown update products, missing component products, invalid option mappings, component quantity above 2000, and `quantityOption` values with fewer than two entries. The same fixture confirms `quantity: 0`, blank `quantityOption.name`, and zero quantity option values are accepted when the component option mapping is otherwise valid. The configured 2025-01 live schema rejects `ProductBundleCreateInput.consolidatedOptions` before resolver execution, so local consolidated-option validation is runtime-test-backed rather than live parity-backed for this version.
- `productVariantRelationshipBulkUpdate` stages normalized `ProductVariantComponent` rows keyed by parent variant. Downstream `ProductVariant.requiresComponents` and `ProductVariant.productVariantComponents(...)` resolve from the effective local relationship graph. Captured validation parity covers unknown parent/component variants returning `field: ["input"]`, code `PRODUCT_VARIANTS_NOT_FOUND`, and parent-as-child input returning `field: ["input"]`, code `CIRCULAR_REFERENCE`. Local validation rejects parent rows that specify both `parentProductId` and `parentProductVariantId`, duplicate parent rows, duplicate component IDs within one create set, component quantities outside `1..9999`, and update/remove rows whose component variant is not currently a child of the parent; rejected inputs return user errors without staging the mutation.
- `combinedListingUpdate` stages parent/child combined-listing relationships locally. The parent product receives `combinedListingRole: PARENT`, child products receive `combinedListingRole: CHILD`, and downstream `Product.combinedListing.combinedListingChildren(...)` resolves child products plus the matched parent variant from submitted parent option values. Captured validation parity covers unknown parent products returning code `PARENT_PRODUCT_NOT_FOUND`; the local mutation also rejects non-parent products, missing `optionsAndValues` for add/edit requests, parent-as-child inputs, duplicate child ids, unknown child products, empty selected parent option values, overlong titles, already-staged child products, and edit/remove overlap with the captured public Admin API error codes.
- Product-domain metafields are normalized as owner-scoped records for `Product`, `ProductVariant`, and `Collection` owners. Besides `id`, `namespace`, `key`, `type`, and `value`, hydrated and staged records carry `compareDigest`, `jsonValue`, `createdAt`, `updatedAt`, and `ownerType` for owner-scoped parity. Generic `node(id:)` / `nodes(ids:)` resolves effective product-domain `Metafield` GIDs through the same owner-scoped records; Metafield `definition` still serializes as `null` for product metafield node selections until a product-owned fixture returns nested definition data. Definition records themselves are modeled through the metafields endpoint group.
- Local `metafieldsSet` support covers product, product variant, and collection owners only. It validates the full input batch before replacing each affected owner metafield set, supports compare-and-set through `compareDigest`, rejects stale string tokens with `STALE_OBJECT`, rejects string tokens for absent rows with `INVALID_COMPARE_DIGEST`, treats `compareDigest: null` as a create-only guard, and preserves Shopify-like atomic no-write behavior when any modeled resolver error is returned. For matching staged/effective metafield definitions, it infers omitted input `type`, rejects explicit type mismatches, and applies the shared definition-backed validation rules documented in the metafields endpoint group. Customer, order, draft-order, shop, discount, and other owner families remain scoped to their own endpoint groups or future issues.
- `metafieldsDelete` uses the same product-domain owner scope and returns ordered `deletedMetafields` entries, including `null` for missing namespace/key rows. Downstream `product(id:)`, `productVariant(id:)`, and `collection(id:)` reads expose staged owner-specific singular `metafield(namespace:, key:)` and `metafields` connection results without live writes. Captured parity includes an ordinary operation-name product-owner set/read/delete/read flow so the generic path is not tied to conformance fixture operation names.
- The singular compatibility root `metafieldDelete(input: { id })` resolves staged or LiveHybrid-hydrated owner metafields by Metafield GID, stages the same `(ownerId, namespace, key)` tombstone used by owner-scoped reads, returns `{ deletedId, userErrors: [] }` on success, and returns `{ deletedId: null, userErrors: [{ field: ["id"], message: "Metafield does not exist" }] }` for unknown IDs without appending the unverified Metafield ID to staged resource bookkeeping. Modern public Admin GraphQL schema probes through 2026-04 and unstable expose plural `metafieldsDelete` but not this singular root, so singular compatibility coverage is backed by focused Rust tests rather than Shopify parity evidence.
- Product search uses a Shopify-style parser for local `products`, `productsCount`, and product bulk-query JSONL artifacts. The local store-backed subset covers boolean grouping, quoted values, term negation, bare text/tag terms, case-insensitive `status:` enum values, `vendor:`, `product_type:`, `title:`, `tag:`/`tag_not:`, `sku:`, `published_status:`, `created_at:`, and `updated_at:` filters. Unknown fielded filters are explicit local no-matches instead of silently returning the full local catalog; cold LiveHybrid reads without relevant product overlay state still passthrough to Shopify/cassettes.
- Local `products(...)` connection sorting is applied after query filtering and before `reverse` plus cursor windowing. Store-backed reads support the captured sort keys `TITLE`, `VENDOR`, `PRODUCT_TYPE`, `PUBLISHED_AT`, and `ID`; `UPDATED_AT` sorts by effective `Product.updatedAt`. `RELEVANCE` uses the deterministic staged connection fallback ordering for local draft state because Shopify's scoring is opaque, while captured cold LiveHybrid relevance reads continue to passthrough/replay the recorded Shopify order.
- In Rust LiveHybrid parity, cold `product`, `productByIdentifier`, `products`, and `productsCount` reads use Pattern 1 cassette passthrough when no relevant local product state exists. Product-owned `metafield` / `metafields` selections stay local until the product-metafields parity slice is cassette-migrated. Once staged products, product deletes, or product-owned metafield shells are present, those same roots stay on the local serializer so read-after-write and owner-scoped metafield behavior remains local without runtime Shopify writes.
- `productCreate`, `productUpdate`, and `productSet` stage `category` and `requiresSellingPlan` from the public 2025-01 product input shapes without runtime Shopify writes. `category` is stored in the product category slot and is exposed through downstream `Product.category { id fullName }` reads; the local input reader also understands older wrapper objects with `productTaxonomyNodeId` when the request shape reaches the handler. Live 2025-01 evidence shows `Product.productCategory` and the deprecated category wrapper inputs are not exposed by the public schema, so those wrappers remain compatibility handling rather than public parity targets for this version. Explicit `requiresSellingPlan` input values take precedence over the default `false`; selling-plan group membership overlays are modeled separately and do not currently derive or mutate this product scalar.
- `productCreate.collectionsToJoin` reuses the collection membership staging path used by collection add-products mutations. Joined custom collections are immediately visible through both `Product.collections` and `Collection.products` / `hasProduct` / `productsCount` reads. Public 2025-01 capture showed unknown collection GIDs in `collectionsToJoin` are ignored with empty `userErrors`, so the local handler skips unknown collection IDs rather than rejecting them; smart collections are still rejected locally because manual membership writes are not valid for smart collections.
- Captured productCreate input-field parity lives in `productCreate-category-parity`, `productCreate-requires-selling-plan-parity`, and `productCreate-collections-to-join-parity`. The same recorder stores validation probes showing invalid category GIDs are top-level `INVALID_VARIABLE`, unknown taxonomy category GIDs are top-level `INVALID_PRODUCT_TAXONOMY_NODE_ID`, `category + productType` is accepted by public 2025-01, and unknown `collectionsToJoin` IDs are ignored.
- `productCreate(product.status)` and `productChangeStatus(status:)` validate against the effective public `ProductStatus` enum before staging. Invalid literals return Shopify's top-level `argumentLiteralsIncompatible` error shape; invalid variables return `INVALID_VARIABLE` coercion errors, and neither branch writes to the local mutation log. Public 2025-01 accepts `ACTIVE`, `ARCHIVED`, and `DRAFT`; `UNLISTED` is only accepted on versioned routes where the public schema exposes it, and `SUSPENDED` remains internal-only.
- Live schema introspection confirms product reorder roots for variant order (`productVariantsBulkReorder`), media order (`productReorderMedia`), option/value order (`productOptionsReorder`), and collection product order (`collectionReorderProducts`). Local support covers variant, option/value, and collection product reorder roots with ordered normalized variants, options, option values, and collection membership rows. `productReorderMedia` has a local validation/payload slice but remains outside full support until product media has a store-backed lifecycle model.
- Product media validation roots (`productCreateMedia`, `productUpdateMedia`, `productDeleteMedia`, and `productReorderMedia`) serialize typed `mediaUserErrors.code` values for modeled branches. The checked-in public parity requests select deprecated `userErrors` as Shopify's field/message-only shape while selecting `code` from `mediaUserErrors`. Missing product validation returns `PRODUCT_DOES_NOT_EXIST`; update/delete unknown media ids return `MEDIA_DOES_NOT_EXIST` with Shopify's captured ID-bearing message. Captured 2026-04 behavior for multiple missing update/delete media ids returns one error on `media` / `mediaIds` that comma-joins every missing GID and pluralizes `Media ids ... do not exist`; the single-id branch remains singular. Invalid create-media source URLs return `INVALID`. Captured 2025-01 behavior for `productReorderMedia` accepts an unknown media id by returning an async `Job` and empty error arrays, so the local validation slice does not fabricate a missing-media error for that branch.
- `productVariantsBulkReorder` stages an ordered effective variant list from `ProductVariantPositionInput.position`, resequences every effective sibling variant's `position` scalar to contiguous `1..N`, and exposes that ordering through downstream `product.variants(...)` and `productVariant(id:)` reads without runtime Shopify writes. Captured 2025-01 validation evidence covers invalid positions (`field: ["positions", i, "position"]`, `code: "INVALID_POSITION"`), duplicate variant ids (`field: ["positions"]`, `code: "DUPLICATED_VARIANT_ID"`), and unknown variants (`field: ["positions", i, "id"]`, `code: "MISSING_VARIANT"`), all with no staged reorder.
- `productOptionsReorder` stages the submitted option order and optional option-value order. It remaps variant `selectedOptions`, derives variant titles from the reordered selected-option tuple, and sorts downstream `product.variants(...)` by the reordered option/value sequence. Captured 2025-01 parity covers typed payload validation for unknown option ids/names, unknown option value ids/names, duplicate option/value names, and mixed id/name selectors. Public Admin GraphQL rejects missing oneOf selectors and the internal `position` key at schema validation time before returning `ProductOptionsReorderPayload`; the proxy's inline parser still rejects those shapes locally with the corresponding reorder guardrail codes.
- `productByIdentifier` supports `identifier.id` and `identifier.handle` against effective local product state. `identifier.customId` resolves product metafields backed by an effective PRODUCT metafield definition whose type is `id`; without that definition, snapshot/local reads return Shopify's captured top-level `NOT_FOUND` error, `Metafield definition of type 'id' is required when using custom ids.`
- `productVariantByIdentifier` supports `identifier.id` against effective local variant state. `identifier.customId` returns `null` until unique variant metafield identifier evidence and indexing are modeled.
- Top-level `productVariants` and `productVariantsCount` resolve from effective local product variants. Supported local query terms are `id`, `product_id`, `title`, `sku`, `barcode`, `vendor`, `product_type`, and `tag`, with the shared Shopify-style boolean parser. Local sorting supports the common `ID`, `TITLE`, `SKU`, `POSITION`, and `INVENTORY_QUANTITY` paths, with connection serialization handled by the Rust connection helpers in `src/proxy.rs`.
- `productTags`, `productTypes`, and `productVendors` serialize distinct sorted `StringConnection` values from effective local products. Empty snapshot state returns empty `nodes`/`edges` with false `pageInfo` booleans and null cursors.
- Product tag normalization is fixture-backed by the 2025-01 `productUpdate-tag-normalization` and `tags-normalization-live-parity` captures. `productCreate`, `productUpdate`, `productSet`, `tagsAdd`, and `tagsRemove` trim tag edges, split comma-separated tag strings for tag mutations, deduplicate and remove tags by a case-insensitive identity, and sort returned `Product.tags` by that identity while preserving the first surviving tag's casing and internal whitespace. Shopify returned a top-level `MAX_INPUT_SIZE_EXCEEDED` error for 251 submitted product tags, and a payload `Product tags is invalid` userError for a tag longer than 255 characters; the local handlers mirror those branches without runtime Shopify writes.
- The deprecated `productCreate(input: ProductInput!)` shape remains accepted by public Admin GraphQL 2026-04 even though introspection lists only `productCreate(product:)`. Captured `harry-test-heelo` evidence shows `input.id` returns a payload `userErrors[{ field: ["input"], message: "id cannot be specified during creation" }]` before scalar validation and without staging a product. The same capture shows `input.variants` is rejected before resolver execution because `ProductInput` does not define `variants`; local schema validation mirrors that top-level `argumentNotAccepted` / `INVALID_VARIABLE` branch instead of accepting ignored variant keys.
- Generic `tagsAdd` / `tagsRemove` dispatch by taggable GID type for the modeled owner families: `Product`, `Order`, `DraftOrder`, `Customer`, and `Article`. Successful non-product updates stage into the owning domain store and downstream `order(id:)`, `draftOrder(id:)`, `customer(id:)`, and `article(id:)` reads expose the updated tags without runtime Shopify writes. Well-formed supported GIDs that do not resolve return an HTTP 200 payload with `node: null` and `userErrors[{ field: ["id"], message: "<Type> does not exist" }]`; the no-data branch does not mutate tag state but keeps the original mutation in the commit log. Captured non-product tag mutations split single comma-separated strings and comma-containing list entries, strip blanks, deduplicate additions by case-insensitive identity, and sort returned tag arrays by case-insensitive identity. Customer `tagsAdd` additionally lowercases newly added all-uppercase variants in the captured path, and customer removes are case-insensitive. Order, Article, and DraftOrder captures showed case-insensitive add dedupe/sort but resource-specific remove behavior: removing exact `Red` succeeds, while removing the case variant `red` leaves an existing `Red` tag in place. `DiscountNode` remains unsupported until local discount tag state and the Shopify `discount_tags_api` flag behavior are modeled. GIDs outside the supported taggable set return Shopify's captured top-level `invalid id` / `RESOURCE_NOT_FOUND` shape instead of a product-scoped payload userError.
- The executable 2025-01 live captures for `productChangeStatus`, `tagsAdd`, and `tagsRemove` against `harry-test-heelo.myshopify.com` compare mutation payloads plus downstream product/status/tag-filtered reads. Status-filtered local catalog searches evaluate the effective product status from store state, so `products(query: "status:...")`, `productsCount(query: "status:...")`, and product bulk exports remain aligned with direct `product.status` reads, including Shopify-style lowercase query values such as `status:active`. Product tag mutations no longer maintain a separate stale search-tag index: staged `tagsAdd`, `tagsRemove`, and `productUpdate(tags:)` update effective tags, and local `products(query: "tag:...")` / `productsCount(query: "tag:...")` read from that same effective tag list immediately. The refreshed live status/tag captures retain Shopify's immediate search-index lag as observation data and compare the delayed, index-converged downstream read as the parity target.
- `productSavedSearches` is handled by the saved-search domain model. It returns staged product saved searches with downstream read-after-write behavior and otherwise preserves the captured empty connection shape for product saved searches.
- `productSet` runs local shape guardrails before staging. Captured 2025-01 Shopify evidence returns top-level `MAX_INPUT_SIZE_EXCEEDED` errors, without a `data` envelope, when `input.variants` has more than 2048 entries or a variant's `inventoryQuantities` has more than 250 entries. The local handler mirrors those top-level branches, rejects product option counts above 3, option values above 100 per option, and files above 250 with payload `INVALID_INPUT` userErrors, and returns `PRODUCT_DOES_NOT_EXIST` on the referenced input field when `input.id` / `input.productId` resolves to nothing. Captured 2026-04 evidence for the `ResourceSetIdentifierExtension` guardrail rejects `identifier:` plus `input.id` before staging with `ID_NOT_ALLOWED`, field `["input"]`, and message `The id field is not allowed if identifier is provided.` Suspended effective products are rejected locally with `INVALID_PRODUCT` on `["input"]`; public conformance setup cannot create suspended products, so that branch is covered by focused Rust runtime tests rather than a live capture.
- `productSet(input:, synchronous: false)` records a local `ProductSetOperation`: the mutation returns a `CREATED` operation with `product: null` and empty `userErrors`, while `productOperation(id:)` can immediately read back the completed operation with `status: COMPLETE`, `product`, and `userErrors`. Unknown operation IDs return `null`, matching the captured helper-root no-data behavior.
- `productDuplicate(productId:, newTitle:, newStatus:)` validates `newStatus` against the public `ProductStatus` enum before staging. When omitted, the duplicate inherits the source product status; when present, the duplicate uses the requested status. Captured parity covers an ACTIVE source with omitted `newStatus` and a DRAFT source duplicated with `newStatus: ACTIVE`, including immediate downstream `product(id:)` status readback.
- `productDuplicate(productId:, newTitle:, synchronous: false)` stages the duplicate product locally, returns the captured async mutation shape with `newProduct: null`, a `ProductDuplicateOperation` in `CREATED` status, and empty mutation-level `userErrors`, then exposes the completed operation through `productOperation(id:)` with `status: COMPLETE`, the source `product`, `newProduct`, and operation `userErrors`. A missing source product follows the 2025-01 `harry-test-heelo` capture: the mutation still returns a created operation with no immediate user errors, while the completed operation carries `userErrors[{ field: ["productId"], message: "Product does not exist" }]` and no products. Local completion is immediate on the next helper read rather than time-delayed, and no supported runtime duplicate path writes upstream.
- `productDelete(input:, synchronous: false)` stages a `ProductDeleteOperation` and tombstones the product in the local store immediately. The mutation returns `deletedProductId: null`, a selected `shop`, a `CREATED` operation, and empty `userErrors`; immediate `product(id:)` reads return `null`, matching the refreshed 2026-04 capture where Shopify completed the async delete before the follow-up read. A second pending async delete for the same product still returns `productDeleteOperation: null` with the captured public userError shape `{ field: null, message: "Another operation already in progress. Please wait until current one is finished." }`. The local helper read exposes the operation as completed with `deletedProductId`, and no supported runtime delete path writes upstream.
- `productDuplicateJob(id:)` models the captured unknown-job read shape as `{ id, done: true }`. Current async `productDuplicate` evidence uses `ProductDuplicateOperation` through `productOperation(id:)`; the job root remains a compatibility helper for the older captured `ProductDuplicateJob` shape.
- `productResourceFeedback(id:)` returns `null` for absent local feedback and returns locally staged feedback after `bulkProductResourceFeedbackCreate`. Product feedback records are keyed by product id and expose `productId`, `state`, `messages`, `feedbackGeneratedAt`, and `productUpdatedAt` without runtime Shopify writes.
- `bulkProductResourceFeedbackCreate` stages `ProductResourceFeedback` records locally for effective active products and returns missing-product userErrors for products the proxy can prove are missing, tombstoned, or non-active without staging partial feedback. The userError field is rooted at the feedback entry's product id (`["feedback", i, "productId"]`) and uses Shopify's captured `Product does not exist` message. Current-channel publication scoping is not inferred unless the runtime has a modeled channel/publication basis for the product. `shopResourceFeedbackCreate` stages shop-level `AppFeedback` in meta state and returns the mutation payload locally. Live probes against `harry-test-heelo` returned top-level `ACCESS_DENIED` for both feedback mutations because the app lacks `write_resource_feedbacks` and Storefront API / Sales Channel configuration, so live success parity is blocked by app configuration rather than local runtime behavior. Strict live parity covers schema-level invalid `ResourceFeedbackState` errors for both feedback mutation roots. Runtime tests guard product feedback success, missing-product validation, downstream productResourceFeedback reads, and shop feedback staging.
- Feedback mutations validate the schema-backed `ResourceFeedbackState` enum, require at least one message for `REQUIRES_ACTION`, reject messages over 100 characters, reject invalid or future `feedbackGeneratedAt` values, and cap bulk product feedback batches at 50 entries. Bulk product feedback input-field errors use indexed `["feedback", i, ...]` fields; the batch-cap userError uses field `["feedback"]`, message `Feedback cannot contain more than 50 entries`, and code `TOO_LONG`. These validation guardrails are covered by focused Rust tests because live resolver capture is blocked by the same resource-feedback app configuration constraints.
- Collection records carry aggregate publication target ids alongside product publication ids. A staged `collectionCreate` starts unpublished; collection publication counts and `publishedOnPublication(publicationId:)` remain unpublished until a local publish mutation adds a target.
- `publishedOnCurrentPublication` is not inferred from aggregate collection publication count. Captured Online Store publishable writes leave it false when the app current publication is not the target.
- Local `publishablePublish` and `publishableUnpublish` currently stage Product and Collection publishables. Product-scoped generic publishable payloads also serialize the selected `shop.publicationCount` from the effective local publication catalog, so explicit publication targets can add derived catalog rows when no publication fixture was seeded. The internal current-channel placeholder used by `publishablePublishToCurrentChannel` updates product aggregate/current-publication state but does not become a shop-level publication row; captured parity shows Shopify keeps `shop.publicationCount` tied to the shop publication catalog. Broader publishable implementers remain unsupported in their own groups.
- Shopify's current docs mark `productPublish` as deprecated in favor of `publishablePublish`; the local product-specific roots remain supported because captured parity fixtures still exercise the deprecated product payload shape used by existing apps. `PublicationInput.publishDate` is accepted as a target field but only online-store channels support future publishing in Shopify, and the field has no effect for unpublish mutations. Subscription-only products (`requiresSellingPlan: true`) are publication-restricted to online-store targets in Shopify; the local aggregate model should not generalize non-online-store success without additional live evidence.
- `productPublish` and `productUnpublish` validate submitted `publicationId` / legacy `channelId` targets against the effective publication catalog before staging. Snapshot/local state with no known publication rows treats arbitrary publication/channel GIDs as missing; LiveHybrid hydrates the publication catalog upstream when it is unknown, then returns Shopify's captured `Publication does not exist or is not publishable` / `Channel does not exist or is not publishable` userErrors for targets still absent from the observed catalog.
- Top-level `publication(id:)`, `publications(...)`, `publicationsCount(...)`, `publishedProductsCount(publicationId:)`, and deprecated `channel` / `channels` roots resolve from the normalized publication catalog in snapshot/local overlay paths. Empty snapshot state returns `publication: null`, `channel: null`, empty channel/publication connections, and exact zero counts without upstream access. Captured 2025-01 `productVariantsBulkCreate` evidence covers explicit `price: null` (`Price can't be blank`), negative and too-large `price`, too-large `compareAtPrice`, negative and too-large inventory-item cost and weight, out-of-range `inventoryQuantities[].availableQuantity`, the 50,000 cumulative `inventoryQuantities` entry cap across the submitted batch, the per-variant inventory location entry cap from `shop.resourceLimits.locationLimit` (200 on the conformance store), 256-character `inventoryItem.sku` and `barcode`, 256-character option values, structured `optionValues` entries that specify both `optionId` and `optionName` or both value `id` and `name`, duplicate option-value tuples in the same create request, and the 2048-item `variants:` GraphQL input cap. Captured cost bounds reject `inventoryItem.cost < 0` and `inventoryItem.cost >= 1000000000000000000`: bulk create/update return a parent `field: ["variants", index]` error plus a nested `field: ["variants", index, "inventoryItem", "cost"]` error, with negative cost using `GREATER_THAN_OR_EQUAL_TO` and over-limit cost using `INVALID_INPUT`; `productSet` returns the nested `field: ["input", "variants", index, "inventoryItem", "cost"]` error with `INVALID_VARIANT`. The captured per-variant location cap branch returned `field: ["variants", "0"]`, `code: "TOO_MANY_INVENTORY_LOCATIONS"`, and counted repeated entries for one valid live location toward the limit. The public 2025-01 and 2026-04 schema rejects root-level `variants[].options` on `ProductVariantsBulkInput` before resolver execution, so the proxy returns the captured top-level `INVALID_VARIABLE` error for that shape rather than a mutation `userErrors` branch. The same live capture showed omitted `price` is accepted for this bulk create path, so the local validator rejects explicit null while preserving Shopify's omitted-price behavior. Rejected scalar, option, and inventory-cap branches return captured `field`, `message`, and `code` values and leave local variant state unchanged. The live 2025-01 `ProductCreateInput` shape has `productOptions` but no `variants` field, so local `productCreate` keeps Shopify's option-backed initial variant behavior and price setup flows through variant mutations.
- `productVariantsBulkCreate`, `productVariantsBulkUpdate`, the local legacy single-variant compatibility roots, and `productSet(input.variants)` stage the extended public variant surface without runtime Shopify writes. Local variant records now round-trip `taxCode`, output `position`, `requiresComponents`, `showUnitPrice`, and `unitPriceMeasurement`; nested `inventoryItem` input stages `requiresShipping`, `countryCodeOfOrigin`, `provinceCodeOfOrigin`, `harmonizedSystemCode`, and `measurement.weight`; and variant `metafields` are stored in the owner-scoped metafield store with `ownerType: PRODUCTVARIANT`. Downstream `productVariant(id:)`, `product.variants(...)`, `inventoryItem(...)`, and variant `metafield` / `metafields` reads expose those staged values. Captured 2025-01 evidence also showed Shopify normalizes inventory-item harmonized system codes by removing punctuation and spaces, so local staging normalizes inputs such as `1234.56` to `123456`. Public 2025-01 and 2026-04 schema probes do not expose a `position` input on `ProductVariantsBulkInput`; when a legacy/internal request shape reaches the local bulk-update handler with `position`, the compatibility path moves the variant within the product connection and resequences positions, backed by focused Rust runtime tests rather than public parity.
- Public 2025-01 `ProductVariantsBulkInput` rejects root-level `requiresShipping`, `weight`, `weightUnit`, and `harmonizedSystemCode`; the captured public path for those concerns is nested under `inventoryItem` where available. The same public schema rejects `imageId` and `imageSrc` before resolver execution, so local root-field handling for those legacy compatibility keys remains input-parser coverage rather than 2025-01 public parity evidence. Bulk variant media validation now rejects `mediaId` plus `mediaSrc` and multi-value `mediaSrc` with captured Shopify `BadRequest` wording before staging; successful `mediaId` association continues to stage variant-specific media IDs against product media.
- In LiveHybrid, `productVariantAppendMedia` and `productVariantDetachMedia` validate and stage against the locally observed product, variant, and product-media graph without runtime Shopify writes. Captured 2026-04 parity covers variant-media pair count, one media id per pair, empty `mediaIds` lists, duplicate variant entries, cross-product variants/media, invalid non-image media, non-ready media, already-attached variants, and unattached detach targets. Empty per-entry `mediaIds` returns a `BLANK` userError at `["variantMedia", index, "mediaIds"]`; an entirely empty `variantMedia` list still returns success with empty `userErrors` and `productVariants`. Snapshot mode still fails closed for these roots when there is no staged product/media graph to validate against. Product suspension is not represented by public captured product reads, so these roots do not claim `PRODUCT_SUSPENDED` parity until a captured suspended-product state is available.
- Gift-card product defaulting for variant create remains dependent on local `Product.giftCard` support; the current product record does not yet carry that product-level flag, so the variant path does not infer gift-card defaults without explicit variant/inventory input.
- `productVariantsBulkCreate.strategy` standalone-variant behavior is fixture-backed by 2025-01 and 2026-04 `harry-test-heelo` captures. With `strategy: DEFAULT`, Shopify deletes the standalone default `Default Title` variant before creating the submitted variant, but preserves a standalone custom variant created through `productOptionsCreate(variantStrategy: LEAVE_AS_IS)`. Omitting the `strategy` argument applies the same default-standalone removal behavior as explicit `DEFAULT`, backed by `config/parity-specs/products/productVariantsBulkCreate-omitted-strategy-default-standalone.json`. The captured success branches returned empty `userErrors` arrays, so there were no `field`, `message`, or `code` values to model for those success paths.
- Rejected bulk variant create and delete payloads follow the captured null/empty payload shape: create returns `product: null` with `productVariants: []`, update returns the current `product` with `productVariants: null` for variant-level errors and `product: null` for unknown-product errors, and delete returns `product: null` for unknown or non-member variants. Empty create, update, and delete batches are no-op successes; empty update returns the current `product`, `productVariants: []`, and no `userErrors`. All rejected branches leave staged options, variants, inventory items, inventory quantities, and product inventory summaries unchanged.
- Remaining variant/option coverage gaps after the current executable fixtures: GraphQL variable-shape errors for missing non-null fields in `inventoryQuantities` are still left to the GraphQL input contract rather than hand-built by the proxy; option lifecycle validation is limited to captured unknown-product/unknown-option branches plus the CREATE/LEAVE_AS_IS/null strategy success paths; `productVariantByIdentifier(identifier.customId)` still returns `null` until unique variant metafield identifier evidence and indexing are modeled; direct live evidence for legacy `productVariantCreate` / `productVariantUpdate` / `productVariantDelete` remains blocked by schema absence on the captured 2025-01 store, so those compatibility roots continue to rely on bulk-variant evidence.
- Product search syntax is intentionally forgiving like Shopify. Captured malformed-looking search strings against a disposable product returned normal data envelopes with no top-level `errors`; `tag:(value` and `tag:("value"` behaved as literal non-matching tag values, while a bare leading `(` and a dangling trailing `OR` still matched the created product. The local product search parser preserves that field-value open parenthesis behavior for local staged product reads instead of treating it as a grouped expression.

### Unsupported and boundary behavior

- Product-domain support is limited to the roots and owner families listed above. Customer, order, draft-order, shop, discount, and other metafield owners remain scoped to their own endpoint groups or future conformance work.
- Custom-id lookups for `productVariantByIdentifier(identifier.customId)` and `collectionByIdentifier(identifier.customId)` remain unsupported until unique metafield identifier evidence and indexing are modeled.
- Full product-owned media lifecycle support remains partial. `productCreateMedia` stages media nodes with the declared `mediaContentType` for image, video, external-video, and 3D model inputs, and falls back to source/extension inference when local compatibility traffic omits the public non-null field. `productUpdateMedia` and `productReorderMedia` preserve the staged media subtype for read-after-write projections. `productDeleteMedia`, `productReorderMedia`, and richer product-media side effects remain bounded to the existing validation/local-overlay slice rather than full supported lifecycle coverage; the proxy does not transcode media or fabricate real dimensions, durations, or processing side effects.
- `DiscountNode` tag handling, a product media lifecycle model, full inventory shipment topology, transfer shipment receipt propagation, and product full-sync job completion are explicit gaps rather than supported local lifecycle claims.
- Unsupported product-adjacent roots should remain on the generic unsupported path unless a local lifecycle model, downstream reads, and executable evidence exist.
