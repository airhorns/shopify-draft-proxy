# Products Endpoint Group

The products group is product-first and deep, with registry entries for supported local behavior and explicit unsupported gaps. It covers product roots plus directly related inventory, metafield, collection, publication, tag, helper, feedback, and product-media roots that are modeled as product-owned behavior.

### Behavior notes

- Product feed reads support Shopify-like no-data behavior in snapshot mode and read-after-write behavior for locally staged feeds. Captured 2025-01 `harry-test-heelo` evidence returns `productFeed(id:)` as `null` for an absent feed id and `productFeeds(first:)` as an empty connection with empty `nodes`/`edges`, `hasNextPage: false`, `hasPreviousPage: false`, and null cursors. Locally staged `productFeedCreate` returns an `ACTIVE` `ProductFeed`, `productFullSync` acknowledges known local feeds, and `productFeedDelete` removes staged feeds so downstream `productFeed(id:)` and `productFeeds(...)` return null/empty again; supported feed mutations do not proxy upstream at runtime. The same live store still returns a top-level `NOT_FOUND`, `Unable to find channel for product feed`, for Shopify-side `productFeedCreate(country: US, language: EN)`, so success conformance is documented as channel-configuration-blocked for that shop. HAR-431 adds executable local-runtime parity for the local create/sync/read/delete contract.
- Selling-plan group lifecycle and membership details are documented in `docs/endpoints/selling-plans.md`.
- `Product.sellingPlanGroups`, `Product.sellingPlanGroupsCount`, `ProductVariant.sellingPlanGroups`, and `ProductVariant.sellingPlanGroupsCount` resolve from the staged selling-plan group membership model. Product and variant memberships are tracked separately, matching captured 2026-04 behavior where a group created with `resources.productIds` applies to the product but not automatically to the product variant.
- Product bundle mutations stage normalized bundle products, `ProductBundleOperation` rows, and `Product.bundleComponents` records locally. Captured guardrails still cover `productBundleCreate` with empty components (`productBundleOperation: null`, user error `At least one component is required.`) and `productBundleUpdate` with an unknown product (`productBundleOperation: null`, user error `Product does not exist`). A live HAR-362 success probe with a disposable component product returned a `ProductBundleOperation` in `CREATED` status with `product: null`, confirming Shopify's asynchronous shape; local staging exposes the product immediately so downstream read-after-write tests can inspect the modeled bundle component graph without runtime Shopify writes.
- `productVariantRelationshipBulkUpdate` stages normalized `ProductVariantComponent` rows keyed by parent variant. Downstream `ProductVariant.requiresComponents` and `ProductVariant.productVariantComponents(...)` resolve from the effective local relationship graph. Captured validation parity covers unknown parent/component variants returning `field: ["input"]`, code `PRODUCT_VARIANTS_NOT_FOUND`.
- `combinedListingUpdate` stages parent/child combined-listing relationships locally. The parent product receives `combinedListingRole: PARENT`, child products receive `combinedListingRole: CHILD`, and downstream `Product.combinedListing.combinedListingChildren(...)` resolves child products plus the matched parent variant from submitted parent option values. Captured validation parity covers unknown parent products returning code `PARENT_PRODUCT_NOT_FOUND`.
- Product-domain metafields are normalized as owner-scoped records for `Product`, `ProductVariant`, and `Collection` owners. Besides `id`, `namespace`, `key`, `type`, and `value`, hydrated and staged records carry `compareDigest`, `jsonValue`, `createdAt`, `updatedAt`, and `ownerType` for owner-scoped parity. Generic `node(id:)` / `nodes(ids:)` resolves effective product-domain `Metafield` GIDs through the same owner-scoped records; Metafield `definition` still serializes as `null` for product metafield node selections until a product-owned fixture returns nested definition data. Definition records themselves are modeled through the metafields endpoint group.
- Local `metafieldsSet` support covers product, product variant, and collection owners only. It validates the full input batch before replacing each affected owner metafield set, supports compare-and-set through `compareDigest`, treats `compareDigest: null` as a create-only guard, and preserves Shopify-like atomic no-write behavior when any modeled resolver error is returned. For matching staged/effective metafield definitions, it infers omitted input `type`, rejects explicit type mismatches, and applies the fixture-backed `max` and `regex` validations represented in the local definition model. Customer, order, draft-order, shop, discount, and other owner families remain scoped to their own endpoint groups or future issues.
- `metafieldsDelete` uses the same product-domain owner scope and returns ordered `deletedMetafields` entries, including `null` for missing namespace/key rows. Downstream `product(id:)`, `productVariant(id:)`, and `collection(id:)` reads expose staged owner-specific singular `metafield(namespace:, key:)` and `metafields` connection results without live writes.
- Product search uses the shared Shopify-style search parser. Endpoint-specific product behavior includes boolean grouping, quoted values, field comparators, simple term-list searches, variant search terms, sort keys, and captured connection cursor/pageInfo baselines.
- In Gleam LiveHybrid parity, cold `product`, `productByIdentifier`, `products`, and `productsCount` reads use Pattern 1 cassette passthrough when no relevant local product state exists. Product-owned `metafield` / `metafields` selections stay local until the product-metafields parity slice is cassette-migrated. Once staged products, product deletes, or product-owned metafield shells are present, those same roots stay on the local serializer so read-after-write and owner-scoped metafield behavior remains local without runtime Shopify writes.
- Live schema introspection confirms product reorder roots for variant order (`productVariantsBulkReorder`), media order (`productReorderMedia`), option/value order (`productOptionsReorder`), and collection product order (`collectionReorderProducts`). Local support covers all four reorder roots with ordered normalized variants, media, options, option values, and collection membership rows.
- `productVariantsBulkReorder` stages an ordered effective variant list from `ProductVariantPositionInput.position` and exposes that ordering through downstream `product.variants(...)` and `productVariant(id:)` reads without runtime Shopify writes.
- `productOptionsReorder` stages the submitted option order and optional option-value order. It remaps variant `selectedOptions`, derives variant titles from the reordered selected-option tuple, and sorts downstream `product.variants(...)` by the reordered option/value sequence.
- `productReorderMedia` stages `MoveInput` media moves, returns an async-style `Job`, and exposes reordered `product.media(...)` and `product.images(...)` connections without runtime Shopify writes.
- `productVariantAppendMedia` and `productVariantDetachMedia` stage variant-specific media ID associations without duplicating or deleting product-level media. Downstream `productVariant.media(...)` resolves from the variant's ordered media IDs against the effective product media set.
- `productByIdentifier` supports `identifier.id` and `identifier.handle` against effective local product state. `identifier.customId` resolves product metafields backed by an effective PRODUCT metafield definition whose type is `id`; without that definition, snapshot/local reads return Shopify's captured top-level `NOT_FOUND` error, `Metafield definition of type 'id' is required when using custom ids.`
- `productVariantByIdentifier` supports `identifier.id` against effective local variant state. `identifier.customId` returns `null` until unique variant metafield identifier evidence and indexing are modeled.
- Top-level `productVariants` and `productVariantsCount` resolve from effective local product variants. Supported local query terms are `id`, `product_id`, `title`, `sku`, `barcode`, `vendor`, `product_type`, and `tag`, with the shared Shopify-style boolean parser. Local sorting supports the common `ID`, `TITLE`, `SKU`, `POSITION`, and `INVENTORY_QUANTITY` paths, with connection serialization delegated to `src/shopify_draft_proxy/proxy/graphql_helpers.gleam`.
- `productTags`, `productTypes`, and `productVendors` serialize distinct sorted `StringConnection` values from effective local products. Empty snapshot state returns empty `nodes`/`edges` with false `pageInfo` booleans and null cursors.
- Product tag normalization is fixture-backed by HAR-583's 2025-01 `productUpdate-tag-normalization` capture. `productCreate`, `productUpdate`, `productSet`, `tagsAdd`, and `tagsRemove` trim tag edges, deduplicate and remove tags by a case-insensitive identity, and sort returned `Product.tags` by that identity while preserving the first surviving tag's casing and internal whitespace. Shopify returned a top-level `MAX_INPUT_SIZE_EXCEEDED` error for 251 submitted product tags, and a payload `Product tags is invalid` userError for a tag longer than 255 characters; the local handlers mirror those branches without runtime Shopify writes.
- HAR-427 refreshed the executable 2025-01 live captures for `productChangeStatus`, `tagsAdd`, and `tagsRemove` against `harry-test-heelo.myshopify.com`. The state mutation specs compare mutation payloads plus immediate downstream product/status/tag-filtered reads. Status and tag search are modeled as eventually consistent indexes: direct `product.status` / `product.tags` changes are visible immediately, status-filter and tag-filter catalog searches can keep using the pre-mutation indexed values during the lag window, and returned product nodes still serialize the current product fields.
- `productSavedSearches` is handled by the saved-search domain model. It returns staged product saved searches with downstream read-after-write behavior and otherwise preserves the captured empty connection shape for product saved searches.
- `productSet` runs local shape guardrails before staging. Captured 2025-01 Shopify evidence returns top-level `MAX_INPUT_SIZE_EXCEEDED` errors, without a `data` envelope, when `input.variants` has more than 2048 entries or a variant's `inventoryQuantities` has more than 250 entries. The local handler mirrors those top-level branches, rejects product option counts above 3, option values above 100 per option, and files above 250 with payload `INVALID_INPUT` userErrors, and returns `PRODUCT_DOES_NOT_EXIST` on the referenced input field when `input.id` / `input.productId` resolves to nothing. Suspended effective products are rejected locally with `INVALID_PRODUCT` on `["input"]`; public conformance setup cannot create suspended products, so that branch is covered by focused Gleam runtime tests rather than a live capture.
- `productSet(input:, synchronous: false)` records a local `ProductSetOperation`: the mutation returns a `CREATED` operation with `product: null` and empty `userErrors`, while `productOperation(id:)` can immediately read back the completed operation with `status: COMPLETE`, `product`, and `userErrors`. Unknown operation IDs return `null`, matching the captured helper-root no-data behavior.
- `productDuplicate(productId:, newTitle:, synchronous: false)` stages the duplicate product locally, returns the captured async mutation shape with `newProduct: null`, a `ProductDuplicateOperation` in `CREATED` status, and empty mutation-level `userErrors`, then exposes the completed operation through `productOperation(id:)` with `status: COMPLETE`, the source `product`, `newProduct`, and operation `userErrors`. A missing source product follows the 2025-01 `harry-test-heelo` capture: the mutation still returns a created operation with no immediate user errors, while the completed operation carries `userErrors[{ field: ["productId"], message: "Product does not exist" }]` and no products. Local completion is immediate on the next helper read rather than time-delayed, and no supported runtime duplicate path writes upstream.
- `productDuplicateJob(id:)` models the captured unknown-job read shape as `{ id, done: true }`. Current async `productDuplicate` evidence uses `ProductDuplicateOperation` through `productOperation(id:)`; the job root remains a compatibility helper for the older captured `ProductDuplicateJob` shape.
- `productResourceFeedback(id:)` returns `null` for absent local feedback and returns locally staged feedback after `bulkProductResourceFeedbackCreate`. Product feedback records are keyed by product id and expose `productId`, `state`, `messages`, `feedbackGeneratedAt`, and `productUpdatedAt` without runtime Shopify writes.
- `bulkProductResourceFeedbackCreate` stages `ProductResourceFeedback` records locally for effective products and returns `Product does not exist` userErrors for missing products without staging partial feedback. `shopResourceFeedbackCreate` stages shop-level `AppFeedback` in meta state and returns the mutation payload locally. Live HAR-362 probes against `harry-test-heelo` returned top-level `ACCESS_DENIED` for both feedback mutations because the app lacks `write_resource_feedbacks` and Storefront API / Sales Channel configuration, so live success parity is blocked by app configuration rather than local runtime behavior. HAR-431 adds executable local-runtime parity for product feedback success, missing-product validation, downstream productResourceFeedback reads, and shop feedback staging.
- Collection records carry aggregate publication target ids alongside product publication ids. A staged `collectionCreate` starts unpublished; collection publication counts and `publishedOnPublication(publicationId:)` remain unpublished until a local publish mutation adds a target.
- `publishedOnCurrentPublication` is not inferred from aggregate collection publication count. Captured Online Store publishable writes leave it false when the app current publication is not the target.
- Local `publishablePublish` and `publishableUnpublish` currently stage Product and Collection publishables. Product-scoped generic publishable payloads also serialize the selected `shop.publicationCount` from the effective local publication catalog, so explicit publication targets can add derived catalog rows when no publication fixture was seeded. The internal current-channel placeholder used by `publishablePublishToCurrentChannel` updates product aggregate/current-publication state but does not become a shop-level publication row; captured parity shows Shopify keeps `shop.publicationCount` tied to the shop publication catalog. Broader publishable implementers remain unsupported in their own groups.
- Shopify's current docs mark `productPublish` as deprecated in favor of `publishablePublish`; the local product-specific roots remain supported because captured parity fixtures still exercise the deprecated product payload shape used by existing apps. `PublicationInput.publishDate` is accepted as a target field but only online-store channels support future publishing in Shopify, and the field has no effect for unpublish mutations. Subscription-only products (`requiresSellingPlan: true`) are publication-restricted to online-store targets in Shopify; the local aggregate model should not generalize non-online-store success without additional live evidence.
- Top-level `publication(id:)`, `publications(...)`, `publicationsCount(...)`, `publishedProductsCount(publicationId:)`, and deprecated `channel` / `channels` roots resolve from the normalized publication catalog in snapshot/local overlay paths. Empty snapshot state returns `publication: null`, `channel: null`, empty channel/publication connections, and exact zero counts without upstream access. The existing live publication catalog fixture captures non-empty `publications` id/name/cursor shape; HAR-319 adds runtime-test-backed local evidence for the remaining root family and lifecycle flow.
- `publicationCreate`, `publicationUpdate`, and `publicationDelete` stage normalized Publication rows locally and never perform runtime Shopify writes. This support is intentionally catalog-level: product and collection publication membership still flows through the existing product-specific or generic publishable roots. Deleting a publication strips that target from locally modeled Product and Collection publication IDs so downstream `publishedOnPublication`, publication counts, and `published_status` filters stop seeing the removed target.
- Product handle generation and validation follows the captured product mutation slice: duplicate title-generated handles are de-duplicated, explicit handles are normalized before uniqueness checks, Unicode letters/numbers are preserved, punctuation-only explicit handles fall back into the `product` handle family, explicit collisions return `['input', 'handle']` userErrors, and explicit handles longer than 255 characters return `['handle']` userErrors without staging partial state. The HAR-22 live probe found no product reserved-word rejection for handles such as `admin`, `products`, `collections`, `cart`, `checkout`, or `new`.
- Product option lifecycle staging is fixture-backed for `productOptionsCreate`, `productOptionUpdate`, and `productOptionsDelete`. The current conformance fixtures cover replacing Shopify's default `Title` option with created options, keeping non-variant option values in `optionValues` but out of `values`, renaming and repositioning options, adding/updating/deleting option values, reordering variant `selectedOptions` after option repositioning, and restoring Shopify's default option/variant graph when all custom options are deleted. Local `productOptionsCreate(variantStrategy: CREATE)` also stages variants for the effective option-value Cartesian product, which keeps `values`, `optionValues.hasVariants`, and downstream `product.variants(...)` reads aligned with Shopify's documented CREATE strategy behavior. Captured 2025-01 `harry-test-heelo` evidence for a second CREATE option shows Shopify keeps the existing variants first after filling the new option's first value, then appends the remaining fanout combinations. That shop accepted 110 variants with no `productOptionsCreate.userErrors` (`field`, `message`, and `code` were all absent because the array was empty), which is above Shopify's historical default 100-variant limit and indicates Extended Variants-relevant behavior on this test shop rather than a default-limit rejection. Expected parity differences are limited to generated `ProductOption`, `ProductOptionValue`, and locally generated variant/inventory-item GIDs. Generic `node(id:)` / `nodes(ids:)` support now resolves ProductOption and ProductOptionValue GIDs from the same effective option graph; missing option/value IDs and unsupported adjacent product Node families still return `null` through generic Node dispatch.
- Product create/set option autogeneration is intentionally asymmetric. Captured `productCreate(productOptions:)` evidence shows Shopify creates one initial variant from the first value of each submitted option, keeps extra submitted values in `optionValues`, and leaves those extra values out of `values` with `hasVariants: false`; it does not create the full Cartesian variant matrix. Captured `productSet(input.productOptions)` evidence shows Shopify requires `input.variants` when product options are supplied, returning `product: null` and `userErrors[{ field: ["input","variants"], message: "Variants input is required when updating product options" }]` instead of implicitly generating combinations. Local staging mirrors both branches without runtime Shopify writes.
- Explicit `productOptionsCreate(variantStrategy: LEAVE_AS_IS)` and an explicit null `variantStrategy` both matched Shopify's default branch in the 2025-01 capture: the standalone existing variant was updated to the first created option value, additional option values remained in `optionValues` with `hasVariants: false`, `values` included only the variant-backed value, and `userErrors` was empty.
- Captured option lifecycle validation branches include `productOptionsCreate` with an unknown product (`field: ["productId"]`, `Product does not exist`), `productOptionUpdate` with an unknown option (`field: ["option"]`, `Option does not exist`), and `productOptionsDelete` with an unknown option id (`field: ["options", "0"]`, `Option does not exist`). These branches stage no upstream Shopify writes.
- Top-level `products(query: "published_status:...")` and `productsCount(query: "published_status:...")` apply locally modeled aggregate product publication visibility. A product is treated as published for this filter only when it is `ACTIVE` and has at least one staged or hydrated publication target; `DRAFT` and `ARCHIVED` products remain unpublished even when publication targets are staged. Richer publication graph/detail parity remains limited to the aggregate product fields listed in the validation fixtures.
- Top-level `collections(query: "published_status:...")` applies locally modeled aggregate collection publication state for staged and snapshot reads.
- Product-side collection membership effects are modeled as normalized product collection rows. `collectionCreate(input.products)` stages initial manual collection memberships without upstream writes. A 2025-01 live capture confirms the create payload immediately returns the selected `products` nodes and `hasProduct: true`, but its selected `productsCount` aggregate remains `0` until a downstream collection read returns the actual count. `collectionAddProducts` appends memberships in submitted product order; `collectionAddProductsV2` returns an async-style `Job` and preserves submitted order for explicitly `MANUAL` collections, while retaining the captured non-manual/default insertion behavior from the 2026-04 product-relationship fixture. Manual `collectionReorderProducts` applies moves sequentially, including insertion into the middle of the sort order. `productSet(input.collections)` replaces the product's effective memberships, while `productDuplicate` copies the source product's effective memberships onto the staged duplicate; downstream `product.collections`, `collection(id:)`, collection `products`, `productsCount`, `hasProduct`, and top-level `collections(query: "product_id:...")` reads resolve from the same staged membership rows.
- `collectionReorderProducts.userErrors.code` is modeled for the 2025-10+ error surface documented by Shopify: missing collections return `COLLECTION_NOT_FOUND`, non-manual collections return `MANUALLY_SORTED_COLLECTION`, and malformed or non-member moves return `INVALID_MOVE`. The checked-in success parity fixture remains a 2025-01 capture where the selected error array is empty.
- Product-level variant-derived aggregates are recomputed from the effective local variant graph after product create/update/set, option lifecycle fanout, variant bulk create/update/delete/duplicate, and product-backed inventory quantity mutations. The serializer emits `priceRangeV2`, legacy `priceRange`, `totalVariants`, `hasOnlyDefaultVariant`, `hasOutOfStockVariants`, `tracksInventory`, and `totalInventory` from those stored aggregates instead of synthesizing zeros or leaving stale snapshot values. Money amounts are stored as two-decimal strings and use the effective shop currency when a shop record is available, otherwise the local-only fallback is `USD`.
- `productSet(input.variants[].inventoryQuantities[])` accepts the live Shopify shape with `locationId`, `name`, and `quantity`. Staged create and update flows store those entries as inventory item `inventoryLevels` rows instead of only collapsing them onto the variant. Downstream `product`, `productVariant`, and `inventoryItem` reads expose the location-level `inventoryLevels`, selected `quantities(names: ...)`, aggregate variant `inventoryQuantity`, and product `totalInventory` from the staged graph. Current live evidence uses `name: "available"`; the local row mirrors that quantity into `on_hand` for read parity and leaves `incoming` at `0` unless separately hydrated.
- Product-level inventory aggregates now follow the effective local inventory graph for product/variant mutations. HAR-592 keeps older captured stale-totalInventory scenarios executable with explicit expected differences where the historical cassette observed Shopify's immediate aggregate lag, while new focused parity fixtures assert the recomputed local contract for price-range and out-of-stock behavior.
- `inventoryItems(...)` lists inventory items from the same product-variant-backed inventory graph as `inventoryItem(id:)`; snapshot no-data reads return an empty connection with false page booleans and null cursors. The local search slice covers captured-safe `id`, `sku`, and `tracked` terms and otherwise stays permissive like other early product search slices.
- `inventoryProperties` returns the captured 2025-01 inventory quantity-name catalog: `available`, `committed`, `damaged`, `incoming`, `on_hand`, `quality_control`, `reserved`, and `safety_stock`, including `belongsTo` / `comprises` relationships used by local quantity staging.
- HAR-468 adds 2026-04 inactive inventory-level lifecycle support for product-backed inventory items. `inventoryDeactivate` now marks a level inactive instead of deleting it from local state; default `inventoryItem.inventoryLevels(first:)` reads continue to show only active levels, while `inventoryItem.inventoryLevels(first:, includeInactive: true)`, `inventoryItem.inventoryLevel(locationId:, includeInactive: true)`, and top-level `inventoryLevel(id:)` can expose inactive levels with `isActive: false`. Reactivating the same inventory item/location preserves the deactivated level's quantity rows and returns the same inventory-level id. The executable `inventory-idempotency-directive-lifecycle-2026-04` parity spec replays the captured `inventoryDeactivate`/`inventoryActivate` documents with field-level `@idempotent(key:)` directives. The live 2026-04 schema on `harry-test-heelo` places `includeInactive` on `InventoryItem.inventoryLevel(s)`, not on `InventoryLevel.quantities`.
- `inventoryAdjustQuantities` stages incremental quantity changes over effective product-backed inventory levels without runtime Shopify writes. Captured 2025-01 evidence covers available deltas, non-available `incoming` adjustments with per-change ledger document URIs, invalid quantity names, missing required nested fields, unknown inventory items/locations, app metadata, and the `staffMember` access-denied companion error. The local validator accepts the mutation quantity-name set separately from set semantics: `available`, `on_hand`, `committed`, `damaged`, `incoming`, `quality_control`, `reserved`, and `safety_stock`. Available deltas mirror into `on_hand` changes, update `inventoryItem.variant.inventoryQuantity` immediately, and recompute affected product `hasOutOfStockVariants`, `tracksInventory`, and `totalInventory`; non-available deltas are visible through downstream `inventoryItem.inventoryLevels(...).quantities(names: ...)` reads without changing available/on_hand totals. Captured 2026-04 contract evidence now requires `@idempotent(key:)` and `changes[].changeFromQuantity` on the versioned route before the local staged branch runs. The executable parity targets cover the 2026-04 missing-`changeFromQuantity` adjust branch, successful set-then-adjust writes, and downstream product/variant/item read effects; the non-available branch compares the whole selected downstream product/variant/item payload and only carves out volatile timestamps plus reconstructed inventory-level GIDs.
- `inventorySetQuantities` stages absolute quantity writes over effective inventory item levels without runtime Shopify writes. Captured HAR-568 validation evidence shows the set mutation accepts only `available` and `on_hand`; `damaged` and `committed` return `INVALID_NAME` at `["input", "name"]` with Shopify's exact "available or on_hand" message. Captured 2025-01 evidence showed `ignoreCompareQuantity: true` accepting set writes, returning an `InventoryAdjustmentGroup`, mirroring `available` deltas into `on_hand` changes, accepting direct `on_hand` writes with paired `available` and `on_hand` change rows, immediately updating `inventoryItem.variant.inventoryQuantity`, and producing historical stale-product captures that are now covered by explicit HAR-592 expected differences. Captured 2026-04 evidence rejects `ignoreCompareQuantity`, requires `@idempotent(key:)`, and uses per-quantity `changeFromQuantity` instead of the older compare/ignore shape. Local validation also rejects quantities outside `[-1_000_000_000, 1_000_000_000]`, negative set quantities, and duplicate `inventoryItemId` + `locationId` pairs before staging.
- `inventoryMoveQuantities` stages same-location quantity moves over effective inventory item levels. Captured 2025-01 evidence showed an available-to-damaged move returning two `InventoryChange` rows, keeping `on_hand` unchanged because both names belong to `on_hand`, and updating variant inventory quantity from available totals. HAR-592 also recomputes affected product stock aggregates after moves. Different-location moves, same-name moves, and unsupported ledger-document branches return visible local `userErrors` and do not contact Shopify.
- `inventoryTransfer` and `inventoryTransfers` are modeled for locally staged transfer records. Empty snapshot state returns `inventoryTransfer(id:)` as `null` and `inventoryTransfers(first:)` as an empty connection with false page booleans and null cursors. The current search/sort slice is intentionally narrow: locally staged transfers are listed in staging order with shared cursor pagination.
- Inventory transfer lifecycle mutations stage records in memory and never proxy supported roots at runtime. Draft create/edit/set-items/remove-items/duplicate/delete preserve downstream transfer reads and raw mutation log order. `inventoryTransferCreateAsReadyToShip` and `inventoryTransferMarkAsReadyToShip` mirror the captured 2025-01 inventory effect by moving origin-level `available` quantity into `reserved`; `inventoryTransferCancel` releases that local reservation when canceling a ready transfer. HAR-434 also covers ready-transfer item quantity edits: `inventoryTransferSetItems` reserves or releases the origin-level delta, and `inventoryTransferRemoveItems` releases removed ready quantities so downstream `InventoryItem` available/reserved reads stay consistent.
